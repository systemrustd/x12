# Stage 1 implementation plan — KmsCore extraction + KmsBackendV2 skeleton

Date: 2026-05-16
Status: draft, for codex + author review
Spec: `docs/superpowers/specs/2026-05-15-rendering-model-v2.md` (§ Migration → Stage 1)
Branch: `rendering-model-v2`

## Goal

Land the structural prerequisites for v2 implementation without
changing v1 behavior:

- **Stage 1a** — extract `KmsCore`, a shared struct holding pure
  protocol-bookkeeping state. `KmsBackend` (v1) embeds it. Pure
  field-move refactor. Nothing else changes.
- **Stage 1b** — add `KmsBackendV2`, a sibling `Backend` impl that
  also embeds `KmsCore`. Wire `YSERVER_RENDER_MODEL` env var to pick
  v1 or v2 at startup. v2 boots through capability queries +
  `GetGeometry` / `InternAtom`; the first paint op logs a single
  "v2 not yet implemented" line and returns `Ok(())`.

Stage 1 ships **two independently-verifiable commits in that
order**: 1a first, 1b second. Each compiles, tests-green, and
clippy-clean on its own.

## Non-goals

Deliberately deferred to later stages to keep 1a a pure move and
1b a buildable stub:

1. **Splitting `WindowState` / `PixmapState` / `CursorState` /
   `PictureState`.** Their metadata halves belong in `KmsCore`
   spec-wise, but each currently has a `vk_mirror` /
   `GradientPicture` / Vk-typed field embedded. Per the spec's
   narrow rule ("nothing enters `KmsCore` until its metadata half
   can be separated from its storage/GPU half"), the whole map
   stays in `KmsBackend` until Stage 2 stands up `DrawableStore`
   and v2 actually needs the metadata side. v1 keeps its
   current layout.
2. **Splitting `dri3_sync_resources` / `dri3_xshmfences`.** Keyed
   by client XID (a record), valued by `VkSemaphore` /
   `FenceMapping` (Vk-typed). Same rule — stays in `KmsBackend`.
3. **Touching `HostX11Backend` / `RecordingBackend`.** `KmsCore`
   is shared between `KmsBackend` and `KmsBackendV2` only. Other
   backends are unaffected.
4. **`KmsBackendV2` paint paths.** Stage 1b stubs every paint /
   RENDER / scene method with a once-per-method `warn!` and
   `Ok(())`. The real components (`PlatformBackend`,
   `DrawableStore`, `RenderEngine`, `SceneCompositor`) land in
   Stage 2.
5. **`fb_w` / `fb_h` migration to `PlatformBackend`.** Derived from
   DRM outputs; stays in `KmsBackend` / `KmsBackendV2` for 1a-1b.
   Moves into `PlatformBackend` in Stage 2.
6. **Adding `fb_dimensions` / `randr_outputs` / `take_input_ctx`
   to the `Backend` trait.** Already non-trait methods on
   `KmsBackend`. Stage 1b adds matching methods on `KmsBackendV2`
   and a thin `KmsBackendKind` enum so `lib.rs` can call them
   uniformly before wrapping in `Box<dyn Backend>`. Trait
   widening is unnecessary churn for now.

### `KmsCore` shape

```rust
// crates/yserver/src/kms/core.rs

pub(crate) struct KmsCore {
    // XID + ID maps
    pub(crate) xid_map: HostXidMap,

    pub(crate) window_id: u32,
    pub(crate) root_visual_xid: u32,

    pub(crate) xkb_keymap: XkbKeymap,
    pub(crate) xkb_state: XkbState,

    // Font protocol state
    pub(crate) font_loader: FontLoader,
    pub(crate) fonts: HashMap<u32, FontState>,

    // Root background (pre-paint metadata; the actual paint
    // happens through backend-specific storage)
    pub(crate) bg_pixel: Option<u32>,
    pub(crate) bg_pixmap: Option<PixmapHandle>,

    // Pointer protocol state
    pub(crate) cursor_x: f32,
    pub(crate) cursor_y: f32,
    pub(crate) active_cursor: Option<u32>,
    pub(crate) button_mask: u16,
    pub(crate) prev_pointer_window: Option<u32>,
    pub(crate) pending_pointer_events: Vec<HostPointerEvent>,

    // Default GC state
    pub(crate) current_font: Option<u32>,
    pub(crate) current_function: GcFunction,
    pub(crate) current_foreground: u32,
    pub(crate) current_background: u32,
    pub(crate) current_fill: FillState,
    pub(crate) current_clip: ClipState,

    // SHAPE regions
    pub(crate) shape_bounding: HashMap<u32, Vec<xfixes::RegionRect>>,
    pub(crate) shape_clip: HashMap<u32, Vec<xfixes::RegionRect>>,
    pub(crate) shape_input: HashMap<u32, Vec<xfixes::RegionRect>>,

    // COMPOSITE redirect records
    pub(crate) alias_registry: AliasRegistry,
    pub(crate) host_window_to_backing: HashMap<u32, PixmapHandle>,

    // RENDER glyphset source data (atlas storage is backend-specific)
    pub(crate) glyphsets: HashMap<u32, GlyphSetState>,
}
```

Type definitions that today live on `KmsBackend`'s side of
`backend.rs` move into `core.rs` alongside the struct:

| Type | Current visibility | New visibility | Notes |
|---|---|---|---|
| `FontLoader` | private (`struct`) | `pub(crate)` | accessed by `KmsBackend` |
| `FontState` | private | `pub(crate)` | accessed by `KmsBackend` |
| `GlyphSetState` | `pub(super)` | `pub(crate)` | accessed by `KmsBackend` |
| **`StoredGlyph`** | **private** | **`pub(crate)`** | **Reach-into pattern (`gs.glyphs.get(&id)` → `StoredGlyph`) from `KmsBackend`'s text-rendering paths means this MUST widen to `pub(crate)` — easy to miss because the current decl has no visibility marker** |
| `GlyphSetFormat` | private | `pub(crate)` | enum, used by both `GlyphSetState` and `StoredGlyph` |
| `FreetypeFace` | `pub(super)` | `pub(crate)` | newtype |
| `XkbContext` / `XkbKeymap` / `XkbState` | `pub` | unchanged | already public |
| `AliasRegistry` / `AliasEntry` | `pub` | unchanged | already public; defined in backend.rs |

Compile error if any of these is forgotten — the rename pass
will surface them all. The table is here so they're not
forgotten during the move itself.

`KmsCore::new()` / `KmsCore::for_tests()` constructors carry the
respective initialisation patches today scattered through
`KmsBackend::open_with_commit` and `KmsBackend::for_tests`.

### Refactor pattern

The diff is mechanical:

1. Each moved field reference `self.<field>` becomes
   `self.core.<field>` inside `KmsBackend` impls. The compiler
   enumerates every site.
2. `Backend::xid_map(&self) -> &HostXidMap` returns
   `&self.core.xid_map` instead of `&self.xid_map`. Same for
   `Backend::window_id`, `Backend::root_visual_xid`.
3. External callers (in `crates/yserver/src/kms/vk/*`,
   `crates/yserver/src/kms/scheduler/*`, etc.) that reach into
   `backend.<field>` for a moved field now reach into
   `backend.core.<field>`. The `pub(crate)` visibility on
   `KmsBackend.core` and `KmsCore.<field>` make this a syntactic
   rename.
4. `KmsBackend::open_with_commit` and `KmsBackend::for_tests`
   delegate moved-field initialisation to `KmsCore::new()` /
   `KmsCore::for_tests()`.
5. No new behavior. No accessor delegation methods on
   `KmsBackend` for moved fields — direct `self.core.<field>` is
   cleaner and the diff is reviewable that way.

### Open path

`KmsBackendV2::open(device_path: &str) -> io::Result<Self>` and
`KmsBackendV2::for_tests() -> Self` mirror `KmsBackend`'s open
paths and **share their DRM / KMS / libinput / input-thread init
sequence** by extracting that sequence into shared helpers in
`kms::core` (or a sibling `kms::platform_open` module — see
"Shared open helpers" below). Vulkan / pipelines / scanout pools
/ scheduler init is **skipped** in v2.

### Shared open helpers

To avoid duplicating ~500 lines of DRM/libinput init across
`KmsBackend::open_with_commit` and `KmsBackendV2::open`, factor
the DRM-device-open / output-enumeration / libinput-context-create
sequence into a free function:

```rust
// crates/yserver/src/kms/core.rs (or a sibling kms::platform_open)

pub(crate) struct PlatformInit {
    pub(crate) device: Arc<drm::Device>,
    pub(crate) render_node_fd: Option<OwnedFd>,
    pub(crate) render_node_path: Option<PathBuf>,
    pub(crate) outputs: Vec<OutputLayout>,
    pub(crate) fb_w: u16,
    pub(crate) fb_h: u16,
    pub(crate) input_ctx: Option<crate::input::SendContext>,
    pub(crate) first_pageflip_logged: Vec<bool>,
}

pub(crate) fn platform_init(
    device_path: &str,
    commit: impl FnOnce(...) -> io::Result<()>,
) -> io::Result<PlatformInit> { /* lifted from KmsBackend::open_with_commit */ }
```

`KmsBackend::open_with_commit` and `KmsBackendV2::open` both call
`platform_init`. **This factoring lands in 1b's commit, not
1a's** — see § "Open-question resolutions" #1. 1a stays a pure
field-move for the cleanest bisect boundary; the helper
extraction belongs in 1b where its consumer (`KmsBackendV2`) is
introduced.

### `Backend` trait impl for `KmsBackendV2`

The trait has 118 methods. They split into four behavioural
categories for Stage 1b:

**A. Pure accessor (return value from `core`)** — ~15 methods.
Real impl, reads from `self.core`. E.g. `window_id`,
`root_visual_xid`, `argb_visual_xid`, `argb_colormap_xid`,
`render_opcode`, `xkb_opcode`, `xkb_info`, `composite_opcode`,
`render_format_for_ynest_id`, `xid_map`.

**B. Bookkeeping mutation (mutates `core`, no storage side)** —
~10 methods. Real impl, mutates `self.core`. E.g.
`register_top_level`, `register_subwindow`,
`unregister_host_window`, `update_host_event_mask`,
`xfixes_change_cursor_by_name`, `set_shape_rectangles`,
`xkb_proxy`, `get_atom_name`, `get_keyboard_mapping`,
`get_modifier_mapping`.

**C. Mixed bookkeeping + storage** — ~30 methods. Stage 1b
**stub**: mutate the bookkeeping portion of `self.core` where
trivially possible, log a single `"v2 not yet implemented: <name>"`
warn via the dedup helper, return `Ok(())`. Examples:
`create_subwindow`, `destroy_subwindow`, `map_subwindow`,
`unmap_subwindow`, `configure_subwindow`, `reparent_subwindow`,
`change_subwindow_attributes`, `create_pixmap`, `free_pixmap`,
`open_font`, `close_font`, `create_cursor`, `create_glyph_cursor`,
`free_cursor`, `define_cursor`, `name_window_pixmap`,
`allocate_redirected_backing`, `release_redirected_backing`.

For each, the stub:
1. Inserts/removes the appropriate XID entry in `core` so
   subsequent `Backend::xid_map` lookups + `core` queries see a
   consistent world.
2. Returns an opaque-but-valid handle when one is expected
   (`PixmapHandle::from_raw(next_host_xid())`, etc.).
3. Logs the gap once per method name.

**Special case — `release_redirected_backing` / `free_pixmap`
on a refcount-zero entry**: `alias_registry.decref` returning
`true` is v1's signal to free the underlying `DrawableImage`.
In v2 Stage 1b there is no storage to free (no `DrawableStore`
yet), so the stub must explicitly **drop the host_window_to_backing
entry + log the gap and return `Ok(())`** — not call into any
v1 cleanup path, not silently ignore the `true`. Treat it as a
deliberate no-op with a comment pointing at Stage 2 where
`DrawableStore::retire` takes over.

**D. Paint / RENDER / scene** — ~60 methods. Stage 1b stub: log
the gap once, return `Ok(())`. No `core` mutation needed. Includes
`fill_rect`, `copy_area`, `copy_plane`, `put_image`, `get_image`,
`read_depth1_pixmap`, `poly_line`, `poly_segment`, all `poly_*`,
`fill_rectangle`, `image_text*`, `poly_text*`, every `render_*`
method, every `dri3_*` method.

**E. Lifecycle / poll / dump** — ~5 methods. Real impl mirroring
`KmsBackend`'s shape (poll DRM fd, page-flip handler is a no-op
since v2 never flips, `dump_scanout` writes a black PPM with a
gap log, `mark_dirty` no-op, `maybe_composite` no-op, `ping`
trivial).

### Gap-log helper

```rust
fn log_v2_gap(&self, method: &'static str) {
    if self.logged_gaps.borrow_mut().insert(method) {
        log::warn!("v2: {method} not yet implemented — paint or composite operation skipped");
    }
}
```

Per-method dedup keeps the log honest under a real client. The
test for 1b ("first paint produces a logged gap") asserts the
warn line appears at least once per opcode triggered.

### `KmsBackendKind` enum + `lib.rs` selector

```rust
// crates/yserver/src/kms/mod.rs

pub enum KmsBackendKind {
    V1(KmsBackend),
    V2(KmsBackendV2),
}

impl KmsBackendKind {
    pub fn open_from_env(device_path: &str) -> io::Result<Self> {
        match std::env::var("YSERVER_RENDER_MODEL").as_deref() {
            Ok("v2") => {
                log::info!("yserver: render model = v2 (skeleton; paint paths unimplemented)");
                Ok(Self::V2(KmsBackendV2::open(device_path)?))
            }
            Ok("v1") | Err(_) => {
                log::info!("yserver: render model = v1");
                Ok(Self::V1(KmsBackend::open(device_path)?))
            }
            Ok(other) => Err(io::Error::other(format!(
                "YSERVER_RENDER_MODEL={other:?} — expected 'v1' or 'v2'"
            ))),
        }
    }

    pub fn fb_dimensions(&self) -> (u16, u16) {
        match self { Self::V1(b) => b.fb_dimensions(), Self::V2(b) => b.fb_dimensions() }
    }
    pub fn randr_outputs(&self) -> Vec<RandrOutput> { /* same pattern */ }
    pub fn take_input_ctx(&mut self) -> Option<SendContext> { /* same pattern */ }
    pub fn composite_and_flip(&mut self) -> io::Result<()> { /* same pattern */ }
    pub fn into_dyn_backend(self) -> Box<dyn Backend> {
        match self { Self::V1(b) => Box::new(b), Self::V2(b) => Box::new(b) }
    }
}
```

`lib.rs:169` changes from

```rust
let mut backend = KmsBackend::open(&device_path)?;
```

to

```rust
let mut backend = KmsBackendKind::open_from_env(&device_path)?;
```

Subsequent `backend.fb_dimensions()`, `backend.randr_outputs()`,
`backend.take_input_ctx()` calls work via the enum. The final
`Arc<Mutex<dyn Backend>>` wrap uses `backend.into_dyn_backend()`.

## Risks

**Risk 1 — `fonts: HashMap<u32, FontState>` move depth.** `FontState`
contains `face: RefCell<FreetypeFace>`. `FreetypeFace` is a
newtype wrapper over `freetype::Face` (no Vk). Move is safe. But
verify no `vk::` reference has crept into `FontState` since the
last read of `backend.rs` — if so, fall back to the narrow rule
and **LEAVE**. (Pre-implementation check: grep `FontState` for
`vk::`.)

**Risk 2 — `glyphsets` move depth.** `GlyphSetState.glyphs:
HashMap<u32, StoredGlyph>`. `StoredGlyph.pixels: Vec<u8>` is
pure CPU bytes (the uploaded-to-atlas copy lives in
`glyph_atlas`). Move is safe. Same pre-implementation grep check
for `vk::` references inside `StoredGlyph` / `GlyphSetState`.

**Risk 3 — Cross-crate / cross-module accessors.**
`crates/yserver/src/kms/vk/*`, `kms/scheduler/*`, etc., may
access `backend.xid_map`, `backend.alias_registry`, etc.
directly. After the move, those sites change to
`backend.core.xid_map` / `backend.core.alias_registry`. Tractable
mechanical rename; compiler enumerates each. (Pre-implementation
check: grep each moved field name across `crates/yserver/src` to
size the blast.)

**Risk 4 — `KmsBackendKind::into_dyn_backend` ownership.**
`lib.rs` holds the backend through several lifecycle steps
(fb_dimensions, randr_outputs, take_input_ctx) before wrapping
in `Arc<Mutex<dyn Backend>>`. The enum keeps the concrete type
across those calls; the `into_dyn_backend` at the wrap site
matches today's pattern. No new ownership challenge.

**Risk 5 — Hardware smoke deferred to user.** Tonight I'll get
to "tests green + cargo clippy clean" but cannot run
`yserver-xfce-hw` (bare-metal). Both commits go in; user runs
smoke in AM. The spec accepts this — Stage 1 is structural and
v1 should be bit-identical. If smoke surfaces a v1 regression,
the response is a follow-up fix, not a 1a revert (since the move
is mechanical and re-doing it is just as much work).

**Risk 6 — `XkbContext` / `XkbKeymap` / `XkbState` borrow
patterns.** xkbcommon types use `xkbcommon::xkb::Context`
internally and several keyboard methods do simultaneous
`&self.xkb_state` / `&mut self.xkb_state` patterns elsewhere in
the codebase. Moving them under `self.core` keeps the same
borrow structure (single `&mut self.core` borrow), but verify the
compiler agrees. If a stuck split-borrow emerges (e.g. method
borrows `self.core.xkb_state` and `self.scanout_pools`
simultaneously and the compiler can't split), the fix is a
local helper that destructures `self.core` to release the
overlapping borrow — same pattern as today's `let core = &mut
self.core;` splits, just one indirection deeper. Not a model
risk; only a refactor mechanical risk.

**Risk 7 — `for_tests` constructors drifting.**
`KmsBackend::for_tests` is exercised by ~unit tests across the
crate. `KmsBackendV2::for_tests()` needs to be a real, working
test constructor too (the v2 stub-boot test depends on it).
Plan: extract a `KmsCore::for_tests()` helper that returns the
shared protocol-state seed, and both `for_tests` builders call
it.

## Pre-implementation checks (before writing 1a code)

Before the first edit:

1. `grep -rn 'FontState\|StoredGlyph\|GlyphSetState' --include='*.rs' crates/yserver/src` — confirm no `vk::` typed field has crept into the types tagged as MOVE candidates.
2. `grep -rn 'backend\.\(xid_map\|alias_registry\|window_id\|font_loader\|fonts\|glyphsets\|shape_\|host_window_to_backing\|xkb_\|bg_pixel\|bg_pixmap\|cursor_x\|cursor_y\|active_cursor\|button_mask\|prev_pointer_window\|pending_pointer_events\|current_\|top_level_order\|next_host_xid\)' --include='*.rs' crates/yserver/src` — size the cross-module rename.
3. `grep -rn 'KmsBackend' --include='*.rs' crates/yserver/src crates/yserver/tests` — find every site that wraps / constructs / matches on the concrete type, so the `KmsBackendKind` enum + selector doesn't miss a spot in `lib.rs` / `bin/yserver.rs`.

These three greps are read-only and cheap. Resolving any unexpected hit before the move starts avoids a half-done refactor.

## Open-question resolutions (locked in 2026-05-16 author session)

Codex review did not engage substantively in the 2026-05-16 author
session (4 invocation attempts, 4 sessions; codex spent its turn
budget on file reads + reasoning tokens but emitted no
`output_text` message; rate-limit usage climbed from 63% → 73%+
across attempts, suggesting silent throttling or a CLI/model
issue). Falling back to author judgement against the spec:

1. **`platform_init` factoring lands in 1b, not 1a.** 1a stays a
   pure field-move (cleanest bisect boundary). 1b owns both the
   v2 skeleton and the platform_init extraction needed to share
   DRM/output/libinput init between v1 and v2.

2. **`glyphsets` MOVES in 1a.** `StoredGlyph` is dense CPU bytes
   (A8/A1/argb32-extracted-to-A8); the GPU-side atlas
   (`glyph_atlas`) is a separate `KmsBackend` field that stays
   put. Risk 2's pre-implementation grep confirms no Vk type
   transitively. Spec litmus test holds (an all-CPU backend
   would still need `glyphsets` to service `AddGlyphs` →
   `CompositeGlyphs`).

3. **`pub(crate)` fields, no accessor methods.** Matches current
   `KmsBackend` style. Adding ~30 accessor pairs would bloat the
   1a diff and complicate the mechanical `self.field` →
   `self.core.field` rename without buying encapsulation that
   external callers (in `kms/vk/*`, `kms/scheduler/*`) wouldn't
   immediately need anyway.

4. **`KmsBackendKind` lives in `crates/yserver/src/kms/dispatch.rs`
   (new file)**, not `mod.rs`. `mod.rs` is 16 lines today —
   bloating it with a 60+-line enum + impl is wrong. Dedicated
   file gives the v1/v2 selector its own home; `mod.rs` adds
   `pub(crate) mod dispatch; pub(crate) use dispatch::KmsBackendKind;`.

5. **C-stub methods: mutate `core` where trivial, pure `log + Ok`
   where the bookkeeping side is non-trivial.** XID-map insert
   on `create_subwindow` / `create_pixmap` / `create_cursor` /
   `open_font` is trivial and keeps `GetGeometry`-style queries
   consistent. Geometry recording on `configure_subwindow`,
   reparenting, etc. is also trivial. Methods whose v1 impl is
   primarily storage-side (`define_cursor` setting a window
   attribute can be skipped) take the pure-`Ok` path.
