# Stage 4 — re-enable COMPOSITE redirect + COW

Date: 2026-05-17
Status: draft (self-reviewed once, awaiting codex review)
Branch: continues `rendering-model-v2`
Predecessors:

## Goal

Make compositing window managers work on yserver — the original
motivation for the rendering-model-v2 rewrite. By Stage 4 close:

- **mate-session with marco-compositor + xfwm4-with-compositing
  render correctly on bee + fuji.** Drop shadows are alpha
  gradients, not opaque bars. Window borders and decorations
  appear as the compositor draws them. Damage events drive
  compositor repaints with the right region tracking.
- The non-composited path (fvwm3, e16, wmaker, marco-no-comp,
  xfwm4-no-comp, MATE-no-comp) **stays correct** — Stage 3's
  close gate stays met.

This is **already beyond v1's capability**: compositing WMs were
broken on v1 by structural design (the abandoned
`render-convolution-filter` branch's T1-T4 work demonstrated
this couldn't be patched without the rewrite). Reaching the
above is what v2's existence is for.

## Non-goals

- **picom** (any backend). Never worked on v1; not a regression
  to skip. picom-glx needs `GLX_EXT_texture_from_pixmap` (a
  separate substrate per `project_compiz_is_required`) and
  picom-xrender has its own deep RENDER dependencies — both are
  follow-up perf-plan or Compiz-substrate work, not Stage 4
  scope. Pinned for after the bare-functioning compositor case
  is shown to work.
- **Compiz**. Same reason — GLX texture-from-pixmap dependency.
- **rendercheck full pass.** The suite's COMPOSITE-shaped test
  setup may partially unblock after 4c (`name_window_pixmap`
  becoming real); re-evaluate then. Not a closing gate.
- **HW cursor plane / direct scanout / submit aggregation.**
  Stage 5 perf strategies; the spec explicitly allows them as
  `SceneCompositor` strategy plug-ins on top of v2.
- **bee 30-min stability + fuji v1/v2 perf diff.** Deferred to
  Stage 4 *close* — until COMPOSITE flows land, v2 is doing
  more work per frame than v1's per-window-mirror walk and the
  comparison is structurally unfair. With Stage 4 done, v2
  reaches its target compose-shape and an apples-to-apples
  capture becomes meaningful. Capture lives in a follow-up
  session post-close, not blocking Stage 4 itself.
- **Multi-client Automatic redirect** (codex round-6 finding).
  Composite spec allows multiple clients to issue
  `RedirectSubwindows(W, Automatic)` on the same window;
  Xorg represents this as a per-window client list
  (`xserver/composite/compalloc.c:153, :201`) and only
  rejects a second **Manual** redirect with BadAccess.
  yserver-core today stores one `RedirectRecord` per
  `(window, subwindows)` key (`crates/yserver-core/src/server.rs:235`)
  and the handler at `process_request.rs:2644` rejects any
  other owner regardless of mode. Stage 4 keeps that
  single-owner model. Real-world targets (mate + marco,
  xfce + xfwm4) only run one compositor each per session,
  so the multi-client-Automatic shape doesn't actually fire.
  Lifting to a client-list shape would require a
  `RedirectRecord` rework in `yserver-core::server` plus a
  per-list damage fanout — explicit Stage-5-or-later work.
  If a real compositor case surfaces that needs it (e.g.
  a screen-recorder running alongside a WM with both as
  Automatic owners), reopen.

## Spec mapping

From the v2 spec § "Stage 4" + I4 + scene layering + `KmsCore`
scope + `name_window_pixmap` adapter row:

- **I4** — COMPOSITE redirect only changes the **target storage**
  for a window. Manual-mode additionally removes the window from
  server-managed scene composition; Automatic keeps it in.
- **Scene layering item 2** — "top-level windows + descendants
  in z-order — only the non-redirected ones." Read together
  with I4: this means **non-Manual-redirected**; Automatic
  stays in the pass.
- **Scene layering item 3** — COW drawn above all regular
  windows, only if a compositor has called `GetOverlayWindow`.
- **Spec line 285 + 360** — "redirect-state change" is an
  enumerated scene-structure damage source.
- **`Backend` adapter line 537** —
  `name_window_pixmap(host_window)` translates to:
  `self.store.ensure_redirected_backing(window_id) → DrawableId`;
  `self.core.alias_registry.incref(...)`; return.
- **DrawableStore exposes** (line 211–240) — `allocate(..., scene_
  participating)`, `set_scene_participating`, `damage`,
  `peek_presentation_damage`, `ack_presentation_damage`,
  **`set_redirected_target(window_id, backing_id)`** — this is
  the substantive new API.
- **Spec § damage I5** — Manual-redirected backings get
  **protocol damage** (fanout via DamageNotify) but **no
  presentation damage**. Compositors react to the DamageNotify
  by reading the backing via NameWindowPixmap and repainting
  into root or COW.

## Cross-cutting concepts

### Backing storage lifecycle (Manual redirect)

For a window `W` under Manual redirect:

1. Compositor client issues
   `RedirectSubwindows(root, Manual)` or `RedirectWindow(W, Manual)`.
   `yserver-core::process_request` records the entry in
   `server.composite_redirects` then calls the backend.
2. `Backend::allocate_redirected_backing(W, w, h, depth)`:
   - **Idempotent**: if `core.host_window_to_backing.get(W)` is
     `Some(B)`, return `B` unchanged (matches v1, line
     `kms/backend.rs:9576-9588`). Same call coming around twice
     via recovery/retry does not double-incref the registry.
   - Else: allocate backing storage via the v2 pipeline —
     `create_pixmap`-shaped allocation against
     `DrawableStore` with `scene_participating=false`.
     (Auto-mode will flip this to `true` later, in step 4;
     Manual stays `false`.)
   - **Seed the backing — explicit order**:
     a. `engine.copy_area(W_id, B_id, full_extent)` —
        uses raw `DrawableId`s, **before
        `set_redirected_target` flips routing**. Source must
        be W's own storage (not B); ordering this before the
        target flip is the unambiguous way to guarantee that.
     b. For RedirectSubwindows (per-child redirect): each
        direct child of W gets its own backing, and the
        child's `copy_area` step runs independently per child.
        For RedirectWindow (W itself redirected): walk W's
        descendants whose paint will land in B post-redirect,
        and `engine.copy_area(D_id, B_id, full_extent_of_D,
        dst_offset = D_pos_relative_to_W)` for each. This is
        the per-hierarchy seed (Cross-cutting § "Per-hierarchy
        redirect" + "Initial backing content").
   - `core.alias_registry.insert(backing_handle, AliasEntry {
     refcount: 1, w, h, depth })` — the "redirect-active"
     hold (Reason 1 per `AliasEntry` doc).
   - `core.host_window_to_backing.insert(W_xid, backing_handle)`.
   - **`store.set_redirected_target(W_id, Some(B_id))`** so
     future paint resolution against W routes to B. **After
     this, `resolve_paint_target(W_xid)` returns
     `(B_id, (0,0))`; descendants resolve to
     `(B_id, child-offset)`.**
   - Return `backing_handle`.
3. **Mode-specific participation flip** (driven by the
   protocol handler post-allocation, via
   `backend.set_window_scene_participation(W, participating)`
   + a paired flip on B if Automatic):
   - **Manual**:
     - `store.set_scene_participating(W_id, false)` — clears
       unpresented presentation damage on W (existing I5
       transactional rule) and bumps W's epoch.
     - B stays `scene_participating=false` (allocation default).
       The scene doesn't read from B; the compositor does, via
       NameWindowPixmap.
     - `scene.mark_scene_structure_damage_for_rect(W_prev_screen_rect)`
       — see "Concrete scene-structure damage on redirect
       transitions" below.
   - **Automatic**:
     - `store.set_scene_participating(W_id, true)` (no-op if
       already true; idempotent on the mapped case).
     - **`store.set_scene_participating(B_id, true)`** — B
       becomes a damage accumulator because the scene's W-entry
       routes its `storage_id` through `redirected_target` to
       B (see "Scene-entry storage routing for Automatic
       redirect" below).
     - `scene.mark_scene_structure_damage_for_rect(W_screen_rect)`
       — the scene's blit-source identity flipped W's-own-storage
       → B even though W's screen geometry didn't move.
4. Compositor calls `NameWindowPixmap(W)`. v2 backend:
   - `core.host_window_to_backing.get(W_xid)` → `backing_handle`
     (or `NotFound` if W isn't redirected; spec compliance).
   - `core.alias_registry.incref(backing_handle)` — Reason 2
     hold for this alias.
   - **Return the SAME `backing_handle`**; v1 returns the
     existing backing's xid on every call (line 9532-9544),
     and v2 must match for protocol-equivalent behavior.
5. Compositor reads B via `RenderComposite(src=B, dst=root,
   ...)` and writes into root. That paint accumulates
   presentation damage on root (which IS scene-participating),
   and the next compose tick re-blits.

For **Automatic redirect**: steps 1–2 happen as for Manual
(`allocate_redirected_backing` is mode-agnostic — v1 already
factors this way per the trait); step 3's
`set_scene_participating(false)` is **skipped**. The scene
keeps W in its walk; the compositor only observes damage,
doesn't take over presentation.

### Single refcount, two reasons

Per `AliasEntry` doc in `core.rs:602-615` and spec line
175–177 ("all converge on a single refcount per storage"):
the **`AliasRegistry`** keeps **one** counter per backing
pixmap. The two refcount sources are bookkeeping labels in
documentation, not separate counters:

- **Reason 1 (redirect-active)** — seeded with 1 at
  `allocate_redirected_backing`, dropped at
  `release_redirected_backing` / UnredirectWindow.
- **Reason 2 (named alias)** — `+1` on each
  `NameWindowPixmap`, `-1` on each `FreePixmap` of the alias.

`decref` returns `true` when refcount hits zero. v2 backend
on `true`:
- Look up backing's `DrawableId` via `store.lookup(handle.as_raw())`.
- `store.decref(platform, backing_id)`. The store's existing
  retirement (I6a fence + RetireDecision) handles the rest.

This matches v1's `release_redirected_backing` (lines
9559-9564) modulo "v1 frees `DrawableImage`; v2 retires
through `DrawableStore`" (spec line 484-486).

### `ensure_redirected_backing` semantics

The spec uses the name `ensure_redirected_backing` in the
adapter row (line 537), suggesting "lookup or allocate"
idempotent semantics. In practice this is identical to v1's
shape if `allocate_redirected_backing` always runs first (at
RedirectWindow activation) and `NameWindowPixmap` does
lookup-only thereafter. v2 follows the v1 split: explicit
`allocate_redirected_backing` (called by the protocol-side
redirect activation) and `name_window_pixmap` (lookup +
incref). The "ensure" naming in the spec is a single helper
name; the trait split survives.

### Concrete scene-structure damage on redirect transitions

Spec § Composite tick: scene-structure damage rects are the
union of pre-change and post-change positions of involved
entries. For redirect-state transitions:

| Transition | Pre-rect | Post-rect | Damage rect |
|---|---|---|---|
| Unredirected → Manual | W's screen rect (= absolute pos + size from `windows_v2`) | empty (W exits scene) | W's previous screen rect |
| Manual → Unredirected | empty | W's screen rect | W's current screen rect |
| Unredirected → Automatic | W's screen rect (from W's own storage) | W's screen rect (from B) | W's screen rect |
| Automatic → Unredirected | W's screen rect (from B) | W's screen rect (from W's own storage) | W's screen rect |
| Manual → Automatic | empty | W's screen rect | W's current screen rect |
| Automatic → Manual | W's screen rect | empty | W's previous screen rect |

Even though Automatic redirect keeps W in the scene, the
**source storage that the scene blits from changes** (W's
own storage → B for activate, reverse for deactivate). Spec
§Composite tick lists redirect-state changes as a
scene-structure damage source unconditionally; the table
honours that — Automatic transitions ARE scene-structure
events because the scene's blit-source identity flips, even
if the on-screen rect is geometrically stable. Codex round-1
flagged the pre-revision "empty" entry on Unredirected →
Automatic as wrong for this reason.

The rect must intersect each output's geometry to project
correctly (the existing `add_projected_damage` helper from
3f.9 handles output-extent clipping).

### Backings as valid X11 drawables

The backing pixmap is a real X11 Pixmap resource: it has an
xid (minted via `core.next_host_xid`).

**Codex round-1 correction**: the v1 path does NOT insert
the backing into yserver-core's resource table via a
`create_pixmap` piggyback. The core `CreatePixmap` handler
inserts the client resource AFTER the backend call;
`allocate_redirected_backing` runs inside the backend with
no client-side resource creation. The protocol-visible
client Pixmap resource is created by `NameWindowPixmap`,
which calls `backend.name_window_pixmap`, takes the returned
backing handle, and registers it in the client's resource
table as a Pixmap (per Composite spec § NameWindowPixmap
semantics).

So Stage 4 doesn't need any special bridge work for
`DamageCreate(backing_xid)`: the alias's client-side
resource exists as soon as `NameWindowPixmap` runs, and
DAMAGE clients reference it through the alias xid (which
maps to the backing's host_xid → B_id). Compositors that
call `DamageCreate(W)` (the redirected window itself) get
the per-paint fanout via the existing request-layer
`damage_fanout` since paint requests still arrive against
W's xid.

When paint resolves through `resolve_paint_target` and lands
on B, the **protocol-damage fanout** in
`yserver-core::damage_fanout` observes the paint on W's xid
(not B's) so DamageNotify fires for clients watching W. The
fanout layer is request-layer and identical for v1/v2;
verify nothing in the v2 paint sites bypasses it by
resolving the xid too early — the rule of thumb is that
`resolve_paint_target` is a backend-internal step that
shouldn't be visible to upstream of `Backend` trait calls.

### Initial backing content

X11 Composite spec (compositeproto.txt § 3.1, line 44–48):
"The rendering of an entire hierarchy of windows is
redirected to off-screen storage. The pixels of that
hierarchy are available whenever it is viewable." Real
Xorg's implementation (`composite/compalloc.c` ~line 556)
creates the backing pixmap **and copies the existing
parent/hierarchy contents into it** at redirect activation.

**Codex round-1 correction**: an earlier draft claimed
"no copy of W's current storage — v1 doesn't copy either,
compositors trigger their first repaint via the Damage
event chain." That's wrong on two counts: (a) v1's behavior
here isn't a good equivalence target since v1's compositor
support is broken anyway; (b) the Damage chain is not a
correctness substitute — a client may legitimately not
repaint after redirection, in which case the compositor
sees a blank backing forever.

**Decision (ordering unambiguous, codex round-2 fix)**:
`allocate_redirected_backing` issues the seed-copy
**BEFORE** flipping `set_redirected_target`. The copy uses
raw `DrawableId`s on both sides, so it does not consult
`resolve_paint_target` — it reads from W's own storage
(`W_id`) and writes to B's storage (`B_id`). Once the copy
returns, `set_redirected_target(W, Some(B))` flips routing
and subsequent paint resolves to B.

(An earlier draft contradicted itself — the lifecycle
section said "before", the decision section said "after."
Round 2 caught it. Both now say "before"; `engine.copy_area`
explicitly takes raw ids and never goes through the
resolve_paint_target indirection.)

For **per-hierarchy** seed-copy:

- `RedirectWindow(W, mode)`: W is the redirected ancestor.
  Seed = `copy_area(W_id, B_id, W_full_extent)` for W
  itself, plus one `copy_area(D_id, B_id, D_full_extent,
  dst_offset = D_pos_relative_to_W)` per descendant D of W
  whose paint will land in B post-redirect.
- `RedirectSubwindows(W, mode)`: each direct child C of W
  becomes its own redirected ancestor with its own B_C.
  Per-child seed runs the RedirectWindow shape for each C.

The descendant walk is bounded by tree depth at activation
time only — not on the steady-state paint hot path. This
is the one-shot cost called out in the round-1 decision.

### Per-hierarchy redirect (descendants + borders)

Composite spec § 3.1: `RedirectSubwindows(W, mode)` marks
each direct child of W as redirected; the child's hierarchy
(child + grandchildren + ...) paints into the **child's**
backing at child-relative coordinates. `RedirectWindow(W,
mode)` marks W itself; W's hierarchy (W + descendants)
paints into W's backing at W-relative coordinates. Either
way: the **nearest redirected ancestor** of any drawable
owns the backing that drawable's paint lands in.

**Implementation**: `resolve_paint_target` becomes
**ancestor-aware AND coordinate-aware**:

```text
struct PaintTarget {
    id: DrawableId,             // where the paint actually lands
    offset: (i32, i32),         // add to every paint rect's (x, y)
}

// Lives on KmsBackendV2 (per round-4 layering fix); needs both
// self.windows_v2 (parent/position) and the store
// (redirected_target). DrawableStore does NOT take a KmsCore
// reference — the walk is backend-side.
impl KmsBackendV2 {
    fn resolve_paint_target(&self, host_xid: u32) -> Option<PaintTarget> {
        let leaf = self.core.lookup_drawable(host_xid)?;
        // Pixmap targets aren't in `windows_v2` and have no parent
        // chain — short-circuit to the no-redirect identity case.
        // (Pixmaps cannot themselves be redirect targets per
        // Composite spec; only windows can. The store's
        // `redirected_target` accessor returns None for pixmaps.)
        if !self.windows_v2.contains(host_xid) {
            return Some(PaintTarget { id: leaf, offset: (0, 0) });
        }
        let mut cur_xid = host_xid;
        let mut cur_id = leaf;
        let mut offset = (0_i32, 0_i32);
        loop {
            // Check if cur_id is itself redirected.
            if let Some(b_id) = self.store.redirected_target(cur_id) {
                return Some(PaintTarget { id: b_id, offset });
            }
            // Walk up: add cur's (x, y) within its parent.
            let geom = self.windows_v2.get(cur_xid)?;
            match geom.parent {
                // Root reached with no redirected ancestor —
                // identity result on the leaf. Use explicit
                // match, not `?`, so we fall through to a
                // valid PaintTarget rather than short-circuiting
                // the whole Option.
                None => return Some(PaintTarget { id: leaf, offset: (0, 0) }),
                Some(parent_xid) => {
                    offset.0 += i32::from(geom.x);
                    offset.1 += i32::from(geom.y);
                    cur_xid = parent_xid;
                    cur_id = self.core.lookup_drawable(parent_xid)?;
                }
            }
        }
    }
}
```

**Codex round-5 fix**: an earlier pseudocode draft used
`let parent_xid = geom.parent?;` which short-circuits the
whole resolver to `None` for any top-level window (root's
direct children have `parent == Some(root_xid)`, but if any
intermediate `geom.parent` is `None` the `?` poisons the
result). Now the loop explicitly matches `None → return
Some(identity)` so the leaf-with-no-redirected-ancestor case
flows correctly. Pixmaps also get a pre-loop short-circuit
because they aren't in `windows_v2`.

**Codex round-2 correction**: an earlier draft returned
just a `DrawableId`. That's insufficient for child windows
— a child C of redirected W with C-position `(cx, cy)`
relative to W paints a rect `(x, y, w, h)` that must land
at `(cx + x, cy + y, w, h)` in W's backing B. The resolver
returns the offset so paint sites add it to every rect.

**Paint-site impact**: every site listed in 4a swaps the
`store.lookup(host_xid) → DrawableId` shape for
`self.resolve_paint_target(host_xid) → PaintTarget`, then
applies `target.offset` to the paint rect(s) before calling
the engine. The offset is `(0, 0)` in the un-redirected
common case (most paint).

The walk cost is bounded by tree depth (typically ≤ 4 for
real apps). Two alternatives, decided in pre-impl check:

(a) **Per-op tree walk** (above). Simple, correct under any
    reparent / restack timing; small per-paint cost.
(b) **Cached effective_redirect_target on each Drawable**.
    Computed at map/configure/reparent time and propagated
    down the tree. Faster steady-state; complex to keep in
    sync under reparent / mode-flip; stale-cache risk.

Recommendation: ship (a) in 4a, profile if hot, move to (b)
in Stage 5 if the per-op walk shows up.

**Border content**: Composite spec includes the window
border in the redirected hierarchy. v2's window storage
currently doesn't model borders explicitly (border width is
metadata in `WindowGeometryV2`). For Stage 4 close:
honour border-width=0 windows.

**Codex round-2 correction**: an earlier draft said "most
modern apps" use border-width=0. Plan now adds an explicit
verification step at the start of 4c hardware-smoke:

1. Run `just yserver-mate-hw-trace` (Justfile:617).
2. The recipe dumps `mate.xtrace` covering every X11 wire
   request, including all `CreateWindow` calls from marco
   + applications.
3. Grep / parse the trace for `CreateWindow` requests:
   `grep -E "Request\(1\): CreateWindow.*border-width=[1-9]"
   mate.xtrace` should be empty for the windows that get
   `Composite::RedirectWindow` later in the same trace.
   (Note: `x11trace -n` output uses hyphenated `border-width`,
   not underscored — verified against
   `docs/assets/xephyr-xclock-fvwm3-trace.log:130`. Correlate
   the matching `CreateWindow` xids with subsequent
   `RedirectWindow` xids to confirm.)
4. Same step for xfwm4 via the equivalent recipe
   (`yserver-xfce-hw-trace`, Justfile:577) before the 4d
   gate.

Three outcomes drive the close decision:

- **Common case** — border_width is non-zero only on
  **decoration parents** but decorations live in child-window
  frames (the marco / xfwm4 pattern). The redirected leaves
  themselves have border=0. No work needed.
- **Edge case** — border_width is non-zero on a **directly
  redirected window** in either WM. Becomes an explicit
  out-of-scope Stage 4 gap with a `known-issues.md` entry:
  the border doesn't render, the rest of the window does.
  Real fix is post-Stage-4 (depth-aware border-fill pass
  in 4a's `allocate_redirected_backing`).
- **Pathological case** — the WM relies on non-zero borders
  for visual correctness. Unblock Stage 4 close by lifting
  the border-fill into a 4a follow-on substage (4a.1) before
  declaring Stage 4 done.

**Damage on descendant paint**: when a child W' paints
into its parent's backing B, the protocol-damage fanout
fires for the CHILD's xid (W'), not the parent's. Clients
watching the child's Damage object get the event; clients
watching the parent's Damage object don't (matching the
protocol — XDamageCreate on an ancestor doesn't auto-cover
descendants).

### Scene-entry storage routing for Automatic redirect

**Codex round-2 finding** (new gap): Automatic-mode redirect
keeps W in the scene (per I4) but routes paint to B. Without
explicit storage routing in the scene walk, **paint into B
would not drive any scene repaint** — B has
`scene_participating=false` (Manual-backing default), so
`store.damage(B, rect)` would be a no-op, and the scene
would peek W's presentation damage (empty, because paint
never lands in W's own storage anymore) → stale frame
forever.

Two coupled fixes:

1. **Automatic backings get `scene_participating=true`**.
   Manual backings stay `false`. The participation flag
   on B is mode-specific. `allocate_redirected_backing`
   takes no mode (per 4c decision); instead, the protocol
   handler — which has the mode — drives a follow-on
   `store.set_scene_participating(B_id, true)` for the
   Automatic case.

2. **`build_scene` resolves each entry's source-storage
   through `redirected_target`**. When the scene walk
   encounters an Automatic-redirected W, the entry's
   `storage_id` is `B_id`, not `W_id`. Paint into B
   accumulates B's presentation damage; the scene reads
   from B; everything flows correctly.

For Manual mode this distinction doesn't fire: W is filtered
from the walk by `set_scene_participating(W, false)`, so
neither W nor B contribute a scene entry. The external
compositor reintroduces B by reading via NameWindowPixmap and
painting into root (which is what drives the actual flip).

This makes `scene_participating` 4-valued in practice:

| Drawable | scene_participating | In `build_scene` walk |
|---|---|---|
| W (unredirected, mapped) | true | yes; storage_id = W_id |
| W (Automatic-redirected, mapped) | true | yes; storage_id = redirected_target → B_id |
| W (Manual-redirected) | false | no |
| B (Automatic backing) | true | no (not a Window-kind entry; scene reaches it only as W's storage_id) |
| B (Manual backing) | false | no |
| Pixmap (any) | false | no |

## Substages

Four substages mirroring the spec carve.

### 4a — `DrawableStore::set_redirected_target` + paint resolution

**Scope**: storage-side plumbing. No protocol surface change
yet — `allocate_redirected_backing` / `name_window_pixmap`
stubs are still `Err`.

**Adds to `DrawableStore`** (`crates/yserver/src/kms/v2/store.rs`):

- New field on `Drawable`:
  `redirected_target: Option<DrawableId>` (default `None`).
- `pub(crate) fn set_redirected_target(&mut self, window_id:
  DrawableId, backing_id: Option<DrawableId>)`. Setting `Some`
  routes future paint resolution; `None` un-redirects. **No
  side effects on damage / refcount / scene_participating** —
  those flips are the caller's responsibility (4c).
- `pub(crate) fn redirected_target(&self, id: DrawableId)
  -> Option<DrawableId>` — leaf accessor returning the field
  value; used by `build_scene` and by `KmsBackendV2`'s walk.
  **No `KmsCore` dependency**; `DrawableStore` stays purely

The handler-side hook fires (gated on
`supports_redirect_activation`):

- **`RedirectWindow(W, mode)`** activates W itself: one
  `activate_one(W, mode)` invocation. `activate_one(target,
  mode)` is the per-window sequence below.
- **`RedirectSubwindows(W, mode)`** activates **each direct
  child of W**, not W itself. The protocol handler walks
  `state.resources.children(ResourceId(W))`
  (`crates/yserver-core/src/resources.rs:920`) at request
  time and invokes `activate_one(child, mode)` per child.
  (Codex round-4 fix: an earlier draft wrote
  `core.windows_v2.children(W_xid)`, but `windows_v2` is
  backend-private and has no `.children` helper; the
  protocol handler doesn't have access to it. The
  yserver-core `resources` table is the right place.)
  New children created later under W also need activation,
  but **at MapWindow / realize time, not CreateWindow time**.
  Per Composite spec (`compositeproto.txt:44-48`): redirected
  hierarchy pixels are available "whenever it is viewable."
  Xorg attaches the redirect record at CreateWindow
  (`xserver/composite/compwindow.c:586`) but allocates
  / checks the backing on realize via `compCheckRedirect`
  (`xserver/composite/compwindow.c:267`).

  v2 mirrors this:
  - `CreateWindow` does NOT call `activate_one` — the new
    child is created with `redirected_target=None` and
    `scene_participating=true` (default).
  - `MapWindow` / `MapSubwindows` (the protocol-handler side)
    consults the parent's `composite_redirects` entry. If
    the parent has `RedirectSubwindows(mode)` recorded and
    the child is currently unredirected, fire
    `activate_one(child, mode)` **AFTER** the v2 backend's
    `map_subwindow` returns, so the Manual participation
    flip lands last.

  **Codex round-5 finding + round-6 ordering fix**: v2's
  existing `map_subwindow` (`crates/yserver/src/kms/v2/backend.rs:2175`)
  unconditionally flips `scene_participating=true`. Under
  Manual redirect this would undo a
  `set_window_scene_participation(W, false)` set earlier.
  The fix: protocol MapWindow / MapSubwindows handlers run
  the activate path AFTER calling `backend.map_subwindow`,
  so `activate_one` fires `set_window_scene_participation(
  W, false)` LAST and wins over `map_subwindow`'s blind
  `true`. Concrete call sequence:

  ```text
  // In handle_map_window / handle_map_subwindows:
  backend.map_subwindow(W)?;                  // flips W.scene_participating=true
  if let Some(parent_mode) = state.composite_redirects
      .get(&(parent(W), /*subwindows=*/true))
      .map(|r| r.mode)
  {
      if !already_redirected(W) {
          activate_one(W, parent_mode);      // For Manual, drops scene_participating=false
      }
  }
  ```

  `handle_map_subwindows` runs `backend.map_subwindow` per
  child (`crates/yserver-core/src/core_loop/process_request.rs:7927`);
  the post-hook fires per child too. Both `handle_map_window`
  and `handle_map_subwindows` get the same patch.

  (An earlier draft said "activate BEFORE map_subwindow" but
  reasoned "map_subwindow's true happens before activate's
  false" — internally contradictory. Round-6 fix: activate
  AFTER map_subwindow, no contradiction, no v2 backend
  change.)

  Alternative considered: make `map_subwindow` redirect-aware
  (consult `Drawable.redirected_target` and skip the
  unconditional `true` flip). Rejected because it touches a
  hot v2 backend path for an edge case; localising the
  ordering to the protocol handler is cleaner.

  (Codex round-3 finding — the earlier "Redirect{Window,
  Subwindows}(W, mode)" sketch was wrong; for Subwindows each
  child gets its own backing per Cross-cutting §"Per-hierarchy
  redirect".)

`activate_one(target, mode)`:
```
let backing = backend.allocate_redirected_backing(target, w, h, depth)?;
match mode {
    Manual => backend.set_window_scene_participation(target, false)?,
    Automatic => {
        backend.set_window_scene_participation(target, true)?;
        backend.set_backing_scene_participation(backing, true)?;
    }
}
```

- **`UnredirectWindow(W)`** deactivates W: one
  `deactivate_one(W)` invocation.
- **`UnredirectSubwindows(W)`** deactivates each direct
  child of W that's currently redirected: walk children
  and invoke `deactivate_one(child)` per redirected child.

`deactivate_one(target)`:
```
let backing = core.host_window_to_backing.get(target).copied()
    .expect("redirected");
backend.release_redirected_backing(backing)?;  // clears B-side internally
backend.set_window_scene_participation(target, mapped(target))?;
```

- **`DestroyWindow(W)`** on a redirected window: same
  `deactivate_one(W)`; W is then destroyed (W-side flip is
  effectively dropped when W is removed, but the B-side
  clear inside `release_redirected_backing` is still
  required per Item 3).
- **Client disconnect** (`process_disconnect.rs`): walks
  `composite_redirects` for the disconnected client; for
  each, runs `deactivate_one` above.

The `RedirectSubwindows` ↔ "activate each child"
expansion is the protocol-handler's responsibility, NOT
the backend's. `allocate_redirected_backing` is per-window
either way.

`server.composite_redirects`. The backend doesn't see this
today — `allocate_redirected_backing` has no mode parameter.

**Decision** (codex round-1 + round-2 refined): two Backend
methods (both default no-op so v1 / RecordingBackend stay
unchanged):
- `set_window_scene_participation(host_window, participating)`
  — flips W's scene participation. Drives the scene-structure
  damage rect for the window's transition.
- `set_backing_scene_participation(backing, participating)`
  — flips the backing pixmap's scene participation. Needed
  for Automatic mode so B accumulates presentation damage
  (per Cross-cutting § "Scene-entry storage routing for
  Automatic redirect").

The protocol handler (gated on `supports_redirect_activation`)
issues, per redirected target (which is W itself for
`RedirectWindow` and **each direct child of W** for
`RedirectSubwindows` — see §"Scope" above for the expansion):

```
let backing = backend.allocate_redirected_backing(target, w, h, depth)?;
match mode {
    Manual => {
        backend.set_window_scene_participation(target, false)?;
        // B stays scene_participating=false (allocation default)
    }
    Automatic => {
        // target stays scene_participating=true (no-op if already)
        backend.set_window_scene_participation(target, true)?;
        backend.set_backing_scene_participation(backing, true)?;
    }
}
```
And on unredirect:
```
backend.release_redirected_backing(backing)?;  // also clears B-side participation internally (round 3)
backend.set_window_scene_participation(W, mapped)?;
```

Note: `release_redirected_backing` drops B's
`scene_participating` flag internally so an aliased
Automatic backing doesn't keep accumulating damage after
its routing target is gone (codex round-3 finding). The
protocol handler doesn't need a separate
`set_backing_scene_participation(backing, false)` call.

Call sites the protocol layer must hook:
- `Composite::RedirectWindow` / `RedirectSubwindows` —
  activate per the above.
- `Composite::UnredirectWindow` / `UnredirectSubwindows` —
  deactivate per the above.
- **Mode-flip via re-redirect** (Manual ↔ Automatic on the
  same `(window, subwindows)` key by the same client). The
  current handler at `crates/yserver-core/src/core_loop/process_request.rs:2657`
  blindly overwrites `composite_redirects[key]` with the
  new `RedirectRecord`. 4b adds the mode-change branch:

  ```text
  let prev = state.composite_redirects.get(&key).copied();
  state.composite_redirects.insert(key, RedirectRecord { mode, owner });
  if backend.supports_redirect_activation() {
      match prev {
          None => {
              // First-time redirect — just activate.
              activate_each_target(window, subwindows, mode);
          }
          Some(p) if p.mode == mode => {
              // Same owner, same mode — idempotent no-op.
              // (allocate_redirected_backing is itself
              // idempotent on this case per 4b body.)
          }
          Some(p) => {
              // Same owner, different mode — flip W/B
              // participation in place. Backing B is
              // preserved per Xorg's compCheckRedirect
              // (`xserver/composite/compwindow.c:172`):
              // `redirectDraw` is updated but the backing
              // pixmap and its named aliases survive
              // until normal FreePixmap. Composite spec
              // (`compositeproto.txt:80`) explicitly says
              // old named pixmaps remain allocated until
              // FreePixmap.
              flip_each_target_mode(window, subwindows, p.mode, mode);
          }
      }
  }
  ```

  `activate_each_target` and `deactivate_each_target` are
  the per-target walkers (one target for `RedirectWindow`,
  per-direct-child for `RedirectSubwindows`) defined in
  §"Scope" above.

  `flip_each_target_mode(window, subwindows, prev_mode,
  new_mode)` (new): for each redirected target (W itself or
  each direct child of W), look up the existing backing in
  `core.host_window_to_backing`, then flip participation
  only:

  ```text
  fn flip_each_target_mode(target, prev_mode, new_mode):
      let backing = core.host_window_to_backing[target];
      match (prev_mode, new_mode):
          (Manual, Automatic) =>
              backend.set_window_scene_participation(target, true)
              backend.set_backing_scene_participation(backing, true)
          (Automatic, Manual) =>
              backend.set_window_scene_participation(target, false)
              backend.set_backing_scene_participation(backing, false)
          (X, X) => unreachable (handled by prev.mode == mode branch above)
  ```

  **No re-seed on mode-flip** (codex round-6 explicit
  decision): the backing's content is preserved as-is across
  the flip. Re-running `copy_area(W, B)` on Manual →
  Automatic would overwrite the current B (which holds the
  in-flight redirected paint the compositor was sourcing
  from) with whatever W's now-unpainted own storage happens
  to contain — strictly worse than the current backing
  contents. The Automatic → Manual case similarly preserves
  B because the compositor needs to read what's already
  there to decide whether to repaint. The seed-copy only
  runs ONCE, at initial activation (4a/4b), not on any
  subsequent mode-flip.

  **Codex round-5 correction**: an earlier draft did
  `deactivate + reallocate` on mode flip, which dropped the
  backing's xid and broke `NameWindowPixmap` aliases that
  client compositors hold. Xorg preserves the backing
  across mode-flip; v2 matches.
- `DestroyWindow` on a redirected window — protocol handler
  calls `release_redirected_backing` then drops the window.
  W-side participation flip is moot (W is gone) but B-side
  participation **is dropped inside `release_redirected_backing`**
  per the round-3 finding (the alias may keep B alive).
- Client disconnect via `process_disconnect.rs` — already
  iterates `composite_redirects` and calls
  `release_redirected_backing`; both participation flips are
  follow-on calls on the same path.

**Adds to `KmsBackendV2`**:

- `set_window_scene_participation(_, host_window, participating)`:
  - Look up `W_id`.
  - Record the previous on-screen rect (captured BEFORE
    the flip) — needed for the scene-structure damage rect
    on participating=false transitions.
  - `store.set_scene_participating(W_id, participating)`.
  - **Scene-structure damage** — always fires per the
    redirect-transition table in Cross-cutting § "Concrete
    scene-structure damage" (codex round 2):
    - participating=true (un-redirect): rect = W's current
      screen rect.
    - participating=false (Manual activation): rect =
      W's pre-flip rect.
    Fed via `scene.mark_scene_structure_damage_rects(&[rect])`
    (new SceneCompositor method, see additions).
  - `scene.mark_scene_structure_dirty()` is the coarse fallback
    (used today by 3f.9 hooks); rect-precise is preferred but
    the coarse marker is correctness-preserving.
- `set_backing_scene_participation(_, backing, participating)`:
  - Look up `B_id` via `store.lookup(backing.as_raw())`.
  - `store.set_scene_participating(B_id, participating)`. No
    scene-structure damage from this call — the damage from
    a mode-flip is the W-side call's responsibility (the
    blit-source identity flip is geometrically on W).

```text
// When emitting a scene entry for window W (inside the

    ...
});

```

**Implementation note** (codex rounds 4 + 5): the existing
`CompositeDraw` struct (`crates/yserver/src/kms/vk/compositor.rs:68`)
holds **only `image_view`** plus geometry; `sampled_ids` is a

(`crates/yserver/src/kms/v2/scene.rs:252`), indexed in lock-

`sampled_ids` directly. No new field on `CompositeDraw`.

- `peek_presentation_damage(source_id)` reads from B for

- `ack_presentation_damage(source_id, ...)` mirrors.
- `touch_render_fence(source_id, ...)` (the I6a hook from
  3f.9) attaches to whichever storage the scene actually
  read from.

W's transform/geometry/clipping stays on W (the scene
entry's projected bounds are still "where W appears on
screen"); only the storage handle reroutes.

For Manual-mode W: `set_scene_participating(W_id, false)`
filters the entry out before this resolution runs, so the

## Risks + open questions

**Risk 1 — `resolve_paint_target` on `get_image`**. Should
`get_image(W_xid)` after redirect read from B (the routed
backing) or W's original storage? X11 spec: `GetImage` reads
what the X server considers W's content, which under redirect
is B. Code answer: resolve. Compositors call `GetImage(B_xid)`
directly anyway (via `NameWindowPixmap`); resolving on the
window xid is for completeness.

**Risk 2 — Mode plumbing**. **Resolved (codex round 1)**:
option (b) — separate `set_window_scene_participation`
Backend method called by the protocol handler. Allocation
stays mode-agnostic; mode is a scene-policy concern that
already lives in `ServerState::composite_redirects`. See 4c
§"Mode propagation" for the decision details and call-site
enumeration.

**Risk 3 — Multi-output COW extent**. COW is screen-extent
in spec terms. Single-output: `fb_w × fb_h`. Multi-output: a
virtual-screen bounding box. Stage 4 targets the
single-output case (the smoke matrix is one screen each on
bee + fuji). Multi-output COW correctness is a follow-up;
risk-listed not blocking.

**Risk 4 — Multiple compositors holding COW**. spec § Risk 5
notes the refcount handles this. Cleanup must be correct:
each client's disconnect drops its `GetOverlayWindow` holds.
Verify via the `process_disconnect` audit above.

**Risk 5 — Backing `scene_participating=false` from creation**.
Per spec line 214–215, Manual-redirected backings get
`scene_participating=false`. The current 4b plan allocates
via `create_pixmap` which goes through the pixmap path
(already `scene_participating=false` by spec). Confirm via
test — the backing should never accumulate presentation
damage even if a paint happens against its xid directly.

**Risk 6 — Order of clears on redirect transition**. When
flipping `participating=false`, the existing
`set_scene_participating` body (`store.rs:625-635`)
clears unpresented presentation damage and bumps epoch.
This means a paint that landed JUST BEFORE the flip and is
mid-snapshot (peek but not ack) could be lost. Spec § I5
"ack ... gated on the snapshot's epoch" handles this — the
ack becomes a no-op against the new epoch. Confirm by
reading the existing 2e ack path.

**Risk 7 — Compositor reads `B` while another paint is
in-flight on `B`**. The compositor's `RenderComposite(src=B,
...)` issues a `cmd_pipeline_barrier2` if needed; the v2
engine already does this for src/dst pairs in 3c.3
(self-alias scratch). The cross-CB case (paint submitted on
graphics queue, compose-read submitted later on same queue)
is handled by same-queue submission ordering — confirm
nothing in 4b breaks the I6a fence flow.
