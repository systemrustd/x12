## Step 1 â€” Prep: independent bug fixes

Two standalone fixes that don't depend on the mirroring refactor and
reduce risk by shrinking it. Both can land while design review on the
mirroring work continues.

### 1a. `restack_window` stack-mode semantics (task #27)

**Why first**: pre-existing bug surfaced by codex review. `TopIf` /
`BottomIf` / `Opposite` are conditional on current occlusion, not
unconditional raise/lower. Fixing this in isolation gives us a tested
DIX-equivalent restack resolver to wire into Step 4's host-stacking
path without inventing the logic mid-refactor.

**Files**: `crates/yserver-core/src/resources.rs` (the existing
`restack_window`), new unit tests for all five modes.

**Client-visible behaviour change**: this is *not* purely internal â€”
fixing the resolver changes `QueryTree` reply order and
`pointer_target_at()` hit-testing before host stacking is forwarded.
Treat it as a client-visible bug fix. Add tests beyond the resolver:
`QueryTree` order after each mode, and pointer-hit test against the
expected child for `TopIf`/`BottomIf`/`Opposite`. Existing wmaker /
e16 visible smoke still passes (they don't exercise these modes).

### 1b. Minimal visual table + CreateWindow visual/colormap rule (task #28)

**Why first**: avoids a host `BadMatch` storm the moment Step 3 starts
forwarding sub-window CreateWindow with non-default visuals. Modern
GTK3 / Qt / conky clients hit this path. Doing it standalone keeps the
mirroring PRs focused.

**Scope**: a tiny visual table (root visual + one 32-bit ARGB visual,
maybe one 16-bit visual). Add a host-visual-xid field to
`resources::Visual`. Update the CreateWindow handler to choose
`CopyFromParent` only when the child visual matches the parent;
otherwise pass explicit visual + `CWColormap`. Full visual table
mirroring (audit #6) stays deferred.

**Constraints**:
- The setup reply must advertise *exactly* the visuals in the table â€”
  no more, no less. Otherwise a client requests a visual ID that the
  setup reply contained but the table doesn't have, and lookup misses.
- `CreateWindow` must reject an unknown visual ID *locally* with
  client `BadMatch` (don't forward to host and produce an async error).
- Fix `CopyFromParent` resolution to inherit `parent.visual`, not
  always `ROOT_VISUAL`. ARGB-parent / CopyFromParent-child needs to
  produce an ARGB child, not a 24-bit child, otherwise we recreate the
  BadMatch we're trying to avoid.

**Files**: `crates/yserver-core/src/resources.rs` (Visual),
`crates/yserver-core/src/host_x11.rs` (visual probing at init,
`create_window` parameter widening), `crates/yserver-core/src/nested.rs`
(CreateWindow handler).

**Gate**: integration test creating a 32-bit ARGB visual window
doesn't produce a host BadMatch. Test `ARGB-parent â†’ CopyFromParent
child` produces an ARGB host window. Unit: unknown visual ID in
CreateWindow returns client BadMatch and never reaches host. wmaker /
e16 visible smoke still passes (root visual path unchanged).

## Step 2 â€” Foundation: every InputOutput window gets a host xid

**Goal**: make `Window.host_xid` always-Some for InputOutput windows
without yet rewiring drawing handlers or switching to host-driven
Expose. Drawing still uses `top_level_host_target`; the new sub-window
host xids are *allocated but dormant* for draws and Expose. This
decouples resource-model change from behaviour change.

**Scope**:
- `resources::Window.host_xid: Option<u32>` invariant: always Some for
  `class != InputOnly`. Add `host_xid_required()` accessor with
  `.expect()` documenting the invariant.
- Add `border_pixmap_host_xid: Option<u32>` parallel to existing
  `background_pixmap_host_xid`.
- CreateWindow handler: for sub-windows (parent != ROOT), allocate a
  host xid and call `host.create_window(host_parent, ..., event_mask
  = 0)` (see Expose constraint below). Forward `background_pixel`,
  `background_pixmap` (translated), `bit_gravity`, `override_redirect`.
  Honour the visual/colormap rule from Step 1b. **Drop the
  post-CreateWindow `GetInputFocus` sync** in
  `host_x11::create_subwindow`.
- DestroyWindow: cascade host destroy â€” but **unregister the host_xid â†’
  window mapping in resources before issuing host.destroy_window**, so
  stale host events drop instead of misroute (foundation for Step 5).

### Critical Step-2 constraints

1. **Do NOT select host `ExposureMask` on sub-windows yet**. The
   existing synthetic-Expose path in nested.rs gates on
   `host_xid.is_none()`. Once Step 2 sets `host_xid`, that gate flips
   and synthetic Expose stops firing. If host Expose isn't routed back
   yet, mirrored sub-windows go dark on map. Two safe patterns:
   - **Recommended**: keep `event_mask = 0` on host child windows in
     Step 2; keep emitting synthetic Expose. Switch to host
     `ExposureMask` in Step 4 alongside the rest of the host-driven
     window-state forwarding. Update the synthetic-Expose gate from
     `host_xid.is_none()` to a new flag (`uses_synthetic_expose: bool`)
     so the transition is explicit.
   - Alternative: route host Expose through the pump in Step 2 (more
     work; surfaces the pump issue below earlier).

2. **When host `ExposureMask` is eventually selected** (Step 3/4),
   it must be selected on the **pump connection**, not the
   create-window connection. The create-window connection's events
   are read by `read_response()` and skipped â€” they never reach
   `expose_event_fanout`. Add a pump-side
   `register_host_window_exposure(nested_id, host_xid)` or move
   child event selection / xid mapping to the pump connection.

3. **Coordinate invariant at creation, not Step 4**. Host x/y must
   already be `origin - border_width`, and host `border_width` must
   be forwarded. Otherwise Step 3 will start drawing into mis-
   positioned bordered sub-windows. Document the invariant on
   `host_drawable_xid` here.

4. **Container `ConfigureNotify` filter â€” fold from Step 6 into here.**
   Once many host child windows exist, any accidental child
   ConfigureNotify on the pump path can be (mis)interpreted as a
   root RANDR resize. Filter `handle_host_container_resize` strictly
   by `host_xid == container` *before* any sub-window CreateWindow
   forwarding lands.

5. **Host-create failure semantics**. With the GetInputFocus sync
   removed, host `BadMatch` / `BadAlloc` becomes async. Don't leave a
   local Window with a "required" host xid that may not exist:
   - Validate visual / depth / parent locally first; refuse with
     client BadMatch / BadValue before sending to host.
   - Forward to host. If a host error eventually arrives (debug fence
     under a feature flag during testing; absorbed silently in
     production), log it. The local Window may end up with a host xid
     that the host never actually allocated; treat that as a
     soft-failure (subsequent draws to it produce host BadDrawable,
     which we already absorb).
   - Roll back the local CreateWindow on locally-detectable failure.

6. **Block `NameWindowPixmap` on newly mirrored sub-windows until
   Step 5** (or move Step 5 here). Once sub-windows have host xids,
   COMPOSITE clients can name them; today's destroy invalidates
   aliases too early. Either:
   - Reject `NameWindowPixmap(window=mirrored-subwindow)` with client
     `BadValue` until Step 5 lands, or
   - Pull Step 5's NameWindowPixmap retention forward into Step 2.
   Decision: **block now, fix in Step 5** (smaller blast radius).

**Files**: `crates/yserver-core/src/{resources.rs, nested.rs, host_x11.rs}`.

**What still works the same**: drawing on a sub-window goes via
`top_level_host_target` (the host xid on the sub-window is dormant).
Synthetic Expose still fires. Sibling occlusion still fictional.
CopyArea still wrong. Per-window cursors still don't work. *That's
deliberate â€” Step 3 + 4 fix those.*

**Gate**:
- Unit: every CreateWindow allocates a host xid; every DestroyWindow
  unregisters it. Synthetic Expose still fires for newly mapped
  mirrored sub-windows (exactly one Expose, no duplicates).
- Unit: stale host events arriving after DestroyWindow drop silently
  (no panic, no misroute).
- Unit: container ConfigureNotify is treated as root resize; child
  ConfigureNotify is ignored (filter test, fold-in from old Step 6).
- Unit: unknown visual ID at CreateWindow rejected with client
  BadMatch (visual table integration with Step 1b).
- Unit: NameWindowPixmap on a mirrored sub-window rejected with
  client BadValue (lifted in Step 5).
- Performance gate: create + map 50â€“200 nested windows in a unit
  benchmark; fail if it exceeds N ms (set N from a baseline run on
  the GetInputFocus-sync-removed code; the alarm is for regressions
  later, not for the current absolute number). For a 50-widget GTK3
  tree on a local Unix socket, removing the sync is expected to be
  enough; CreateWindow writes pipeline naturally.
- Visible smoke: wmaker / e16 unchanged.

## Step 3 â€” Drawing handlers: convert in batches

**Goal**: drawing on a sub-window targets its own host xid (no
coordinate translation). After this step sibling clipping / occlusion
work, CopyArea returns the visible-only portion.

> **Status: landed in commit 5f1a85e combined with Step 4b.** The
> originally-planned split between Step 3 (drawing rewire) and
> Step 4b (Expose source switch + map + stacking) couldn't actually
> be landed independently â€” the X11 model requires the drawing
> route, host-side mapping, and Expose source to flip together. A
> first attempt at "rewire `top_level_host_target` only" passed solo-
> client smoke (xeyes, xclock, xterm without a WM) but produced
> background-only / text-missing rendering under wmaker. Combining
> 3 + 4b in a single commit, plus three orthogonal fixes that
> showed up during integration, closed the visible smoke. The fixes:
>
> 1. **Cross-connection sync fence after host CreateWindow** (in
>    `host_x11::create_subwindow`). The dispatch layer's
>    pump-connection `ChangeWindowAttributes(ExposureMask)` raced
>    the main connection's CreateWindow under heavy WM activity â€”
>    pump's CWA could arrive at the host before the window existed,
>    BadWindow silently absorbed, ExposureMask never applied, host
>    never sent Expose, client never redrew. The plan's "drop the
>    GetInputFocus sync" was wrong in practice given the dual-
>    connection model; restoring the sync (one round-trip per
>    window create) closes the race.
> 2. **Stop clearing `host_xid` on reparent-away-from-root** (in
>    `resources::reparent_window`). Leftover Phase-2 logic that
>    destroyed the host subwindow when a top-level moved away from
>    root and cleared the local Window's `host_xid`. Step 4a now
>    forwards `XReparentWindow` to the host and the host child
>    stays alive, so clearing the field defeated the per-window
>    drawing route under any WM that reparents (i.e. all of them).
> 3. **Per-client keyboard pump must drop `POINTER_EVENT_MASK`** (in
>    `host_x11::select_keyboard_events`). Latent bug pre-dating
>    this phase but exposed by interactive testing: `ButtonPress`
>    is exclusive in X11, only one client per window. The main
>    `HostInputPump` selects POINTER_EVENT_MASK on the container at
>    startup; each per-client kb pump tried to also select
>    POINTER_EVENT_MASK and got BadAccess for the entire CWA,
>    leaving the connection silent. Selecting only KeyPress |
>    KeyRelease | StructureNotify on the kb pump avoids the
>    collision and keyboard dispatch works.
>
> The original split's intuition (3 = mostly mechanical) was right
> at the per-handler edit level; it underestimated the architectural
> coupling with Step 4 and the cross-connection sync semantics.

Convert every site currently calling `host_drawable_target` â†’
`(target_xid, x_off, y_off)` to call `host_drawable_xid` â†’
`target_xid` (no offsets). The sub-window's host xid (allocated in
Step 2) is now used.

**Use a grep-driven checklist, not the family names.** Run
`grep -n host_drawable_target crates/yserver-core/src/nested.rs` and
work top-to-bottom; the family names below are an organising lens, not
an exhaustive list. Existing audit at the time of writing showed
~25 sites including (incomplete): `PolyFillRectangle`, `PolyRectangle`,
`ClearArea`, `PolyArc`, `PolyFillArc`, `PolySegment`, `PolyLine`,
`PolyPoint`, `FillPoly`, `CopyArea`, `CopyPlane`, `PutImage`,
`GetImage`, `MIT-SHM::PutImage`, `MIT-SHM::CreatePixmap`,
`MIT-SHM::CreateSegment`, `PolyText8/16`, `ImageText8/16`,
`RENDER::CreatePicture`-via-drawable, COMPOSITE
`NameWindowPixmap` source, plus any helper still using
`top_level_host_target`. **Re-grep at the start of each batch** â€”
Phase 3.5 added several since the last full audit.

**Suggested batches** (one PR each, separately verifiable):

- 3a: rectangle / arc family
- 3b: line / segment / point / poly family
- 3c: copy family (CopyArea, CopyPlane)
- 3d: image family (PutImage, GetImage, MIT-SHM paths)
- 3e: text family (PolyText, ImageText)
- 3f: extension paths (RENDER, COMPOSITE drawable resolutions)

**Dual-mode `apply_gc_clip` during the batches**. Don't delete the
`(x_off, y_off)` parameters until *all* call sites are converted.
Converted handlers pass `0, 0` (no offset); old handlers keep passing
the synthetic offset. After the last batch lands, delete the

**Per batch gate**: drawing-on-sub-window unit test for at least one
op in the batch shows the request hitting the sub-window's host xid
with unshifted coordinates. **Pixel/`GetImage` integration test** for
overlapping siblings (rectangle batch) and `CopyArea` from a

After all batches land, **delete** `top_level_host_target` and the

`apply_gc_clip` loses its offset args. (The grep-and-delete step

**Goal**: sibling stacking, per-window cursor, ChangeWindowAttributes

> **Order note:** the post-Step-2 finding (see Step 3 above) means
> the *attribute / geometry / reparent* portions of Step 4 must land
> before Step 3's drawing rewire to keep wmaker visible-smoke green.
> Splitting Step 4 into 4a (attrs + reparent + configure-geometry,
> sub-window host children stay unmapped & dormant for Expose) and
> 4b (stacking + map-on-host + Expose-source switch) is reasonable.
> 4a is enough infrastructure for Step 3's batches to land cleanly.

**Scope**:
- ChangeWindowAttributes: forward bg-pixmap/pixel, border-pixmap/pixel,
  bit-gravity, override-redirect, cursor.
- ConfigureWindow (geometry portion): forward x/y/w/h/border-width
  with the early-out. (Coordinate invariant `origin - border_width`
  was already applied at CreateWindow in Step 2.)
- ConfigureWindow (stacking portion): use the Step-1a resolver to
  produce the final child-list, then mirror to host as Above-then-Below
  walk. Wire in the `sibling_above` cache for no-op suppression.
- ReparentWindow: forward XReparentWindow + re-issue stacking on the
  new parent's children list.
- MapWindow: pre-map stacking refresh + SHAPE no-op stub + XMapWindow.
- UnmapWindow: XUnmapWindow.
- CirculateWindow: forward.
- **Switch sub-window Expose source**: select `ExposureMask` on the
  pump connection for sub-window host xids and route incoming Expose
  through `expose_event_fanout`. Flip `uses_synthetic_expose = false`
  for sub-windows. Verify single-Expose semantics on map and on
  uncover (no duplicate Expose from synthetic + host paths).

**Files**: `crates/yserver-core/src/{nested.rs, host_x11.rs}`.

**Gate**:
- Unit per stack mode + no-op cache hit (extends Step 1a tests).
- Unit: cursor on a sub-window hits XDefineCursor on its host xid.
- Unit: a sub-window receives exactly one Expose on map (no duplicate

- Unit: host WM (kwin / mutter / etc) restacking our top-levels does

## Risks

- **Stacking regression**: Step 4's stacking-mirror is the most likely
  place for a wmaker right-click-delivery-style regression. The
  per-mode unit tests + the explicit visible gate ("wmaker right-click
  works") guard against this.
- **Performance regression from extra host CreateWindow traffic**:
  Step 2 dominates here. Mitigated by dropping the GetInputFocus sync;
  if still slow, consider host request batching.
- **InputOnly skip turns out to matter**: Step 4 should add a regression
  test for InputOnly overlays above InputOutput children before
  shipping. If the test fails, add InputOnly host backing in a small
  follow-up.
- **SHAPE-deferred WM fidelity**: documented expectation; not a
  regression.
- **External host WM restacks our top-levels**: a real WM running on
  the host (kwin / mutter / fvwm) will restack our top-level windows
  when reparenting them into its own frames. The `sibling_above` cache
  in Step 4 must key on intra-ynest order, not host-observed order;
  external host restacks of windows where `host_xid != container`
  must be ignored on the pump path (same filter as the container
  ConfigureNotify guard introduced in Step 2). Test explicitly.
