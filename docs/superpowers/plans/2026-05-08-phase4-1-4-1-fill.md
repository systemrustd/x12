## Architecture

### The render-pass-per-op pattern

Vulkan-direct ops write into a target's mirror `VkImage` by:

1. Lazily transition the mirror from
   `SHADER_READ_ONLY_OPTIMAL` (the layout the composite read /
   mirror upload leaves it in) to
   `COLOR_ATTACHMENT_OPTIMAL`.
2. `vkCmdBeginRendering` with `loadOp=LOAD` (preserve existing
   pixels — solid-fill only writes the rects).
3. Set viewport = mirror extent; scissor = intersection of GC
   clip-rectangles with the mirror extent.
4. Issue the actual op:
   - `PolyFillRectangle` → `vkCmdClearAttachments` with one
     `VkClearRect` per request rectangle, `VkClearAttachment` set
     to the GC's foreground color. **No pipeline / draw call
     required** — `cmd_clear_attachments` is the natural Vulkan
     primitive for a solid-color fill inside a render pass.
   - `ClearArea` → same primitive, color = window's
     `background_pixel` (or transparent if the window has none).
5. `vkCmdEndRendering`.
6. Transition back to `SHADER_READ_ONLY_OPTIMAL` so the next
   composite-pass read sees a layout it can sample.

`cmd_clear_attachments` honours the bound scissor — GC
clip-rectangles fall out of step 3 with no extra wiring.

### Submit cadence

For 4.1.4.1: **one submit + one `vkQueueWaitIdle` per op**. Same
sync model as `MirrorUploader`. Slow per-op (microseconds × number
of fills per frame), correct, simple.

The plan calls for a "per-target batch CB" eventually (design
§"Data flow" — flush at frame boundary / `GetImage` /
destruction). That's deferred: `PolyFillRectangle` + `ClearArea`
together are < 100 ops/frame in any realistic session, and the
sync overhead of one wait_idle per op is dominated by other
backend work (mirror upload also wait_idles).

Batch CB lands as part of 4.1.4.6 RENDER `Composite` (where the
op rate spikes into the thousands and batching becomes
load-bearing).

### Coexistence with still-pixman ops

**Wholesale port** (per user direction): `PolyFillRectangle` and
`ClearArea` no longer call pixman. Pixman code for these two ops
is deleted in this slot.

Two coexistence hazards during 4.1.4 transition (4.1.4.1 …
4.1.4.9):

1. **Mirror-upload clobber.** When a still-pixman op runs on the
   same drawable (`CopyArea`, `PutImage`, lines, text, …), it
   marks the mirror fully damaged. Next frame's `MirrorUploader`
   re-uploads the entire pixman buffer — overwriting any
   Vulkan-direct writes from earlier the same frame. Visible
   symptom: fills/clears flicker or disappear when followed by a
   pixman op on the same drawable.
2. **Stale-pixman read.** Still-pixman ops read the pixman buffer
   as their source. Pixman buffer no longer reflects
   `PolyFillRectangle` / `ClearArea` results. `CopyArea(src=W, …)`
   reading regions of W that we filled via Vulkan reads stale
   bytes.

Both are accepted-risk for the family-by-family port. They
shrink as more ops port, vanishing entirely after 4.1.5.

**Diagnostic plan:** if real-world testing shows visible
flickering or wrong-content artefacts attributable to (1) or (2),
land per-rect damage tracking as a sidequest before continuing
the port — `MirrorDamage` already supports `mark_rect`; the
remaining work is changing `with_image_mut` callers to pass their
write rect rather than `mark_full_damage`.

### GC clip handling

Phase 3.x already parses `SetClipRectangles` into
`KmsBackend::current_clip` and offers
`current_clip_rects_in_dst_space()` for the pixman path. We
re-use that:

- `ClipState::None` → scissor = whole mirror extent (no
  restriction).
- `ClipState::Rectangles` → scissor = each clip rect (multiple
  scissors via `vkCmdSetScissor` with > 1 rect; pipeline must
  declare `multiViewport` if we go above one — we don't here, so
  fall back to issuing the op once per clip rect, or use a single
  scissor = bounding rect of the clip list as a coarsening that
  rendercheck `fill` should still pass).
- `ClipState::Pixmap` → fall back to the (still-running) pixman
  path until 4.1.4.8.

For minimum viable in this slot: use **one scissor = bounding
rect** of the GC clip when `Rectangles`, else the full mirror.
rendercheck `fill` covers the simple case; we tighten in 4.1.4.8
when the pixmap-mask clip lands and we have to revisit anyway.

**Step 1:** Create `vk/ops/fill.rs` with the public surface:

```rust
pub fn record_fill_rectangles(
    vk: &VkContext,
    cb: vk::CommandBuffer,
    target: &mut DrawableImage,
    color: [f32; 4],
    rects: &[vk::Rect2D],
    clip_scissor: vk::Rect2D,
) -> Result<(), vk::Result>;
```

Body sequence: barrier `current_layout → COLOR_ATTACHMENT_OPTIMAL`,
`cmd_begin_rendering` (loadOp=LOAD) on `target.vk_image_view`,
`cmd_set_viewport` (full mirror), `cmd_set_scissor` (clip
intersected with mirror), `cmd_clear_attachments` with the rects
+ color, `cmd_end_rendering`, barrier `COLOR_ATTACHMENT_OPTIMAL →
SHADER_READ_ONLY_OPTIMAL`. Update `target.current_layout`.

**Step 2:** Build, no call sites yet.

### Task 2: Per-op submit helper on `KmsBackend`

**File:** `crates/yserver/src/kms/backend.rs`

A small helper that allocates a one-shot CB (from a backend-level
ops command pool, separate from the `MirrorUploader`'s pool — the
ops CB submits drawing work, the upload CB submits transfer
work; sharing pools risks ordering bugs). Records via the
caller-supplied closure, submits, `vkQueueWaitIdle`s.

**Step 1:** Add `ops_command_pool: Option<vk::CommandPool>` field.
Initialise in `KmsBackend::new` alongside `MirrorUploader`.
Destroy in shutdown.

**Step 2:** Add `with_ops_cb<F: FnOnce(vk::CommandBuffer) ->
Result<...>>(...)` helper that allocates a primary CB,
`begin_command_buffer` (ONE_TIME_SUBMIT), runs the closure,
`end_command_buffer`, submits, `queue_wait_idle`s, frees the CB.

**Step 3:** Build, commit.

```
feat(kms/backend): per-op CB helper for Vulkan drawing-op submits (4.1.4.1)
```

### Task 3: Wire `PolyFillRectangle` → Vulkan path

**File:** `backend.rs` — `poly_fill_rectangle` handler.

**Step 1:** Compute the GC foreground color as `[f32; 4]`
(0xRRGGBB → r/255, g/255, b/255, 1.0).

**Step 2:** Compute the clip scissor: `current_clip_rects_in_dst_space()
→ Vec<Rectangle16>`, intersect / coarsen to a single bounding
`vk::Rect2D` against the mirror extent.

**Step 3:** Convert request rects (`Vec<Rectangle16>`) to
`Vec<vk::Rect2D>`.

**Step 4:** Dispatch:

- If the target has `vk_mirror.is_some()` and the backend has a
  compositor pipeline + ops CB pool: call
  `with_ops_cb(|cb| fill::record_fill_rectangles(...))`. Pixman
  code for fill is **not** called.
- Else: pixman-only (existing behaviour). Falls back when Vulkan
  didn't come up.

**Step 5:** Smoke under `just yserver-fvwm3-xterm-hw` — verify
the fill paths still render correctly (visible regression: solid
fills now go through Vulkan but the pixman shim keeps results
identical).

**Step 6:** Commit.

```

```

## Risks

- **Mirror-upload clobber.** Other-family ops still in pixman
  trigger `mark_full_damage` → mirror upload re-writes the whole
  mirror, including the rect Vulkan just filled. Mitigated by
  the dual-path shim (Vulkan-direct AND pixman fill). Without
  the shim, every cross-family interaction loses Vulkan-direct
  writes — visible as "fills disappear when the next CopyArea
  runs". Drop the shim only at 4.1.5.
- **Layout-tracking divergence.** `DrawableImage::current_layout`
  is updated by both `MirrorUploader` and the new ops path.
  Symptom: `oldLayout` mismatch in the next pipeline barrier →
  validation-layer error or driver crash. Test under
  `RUST_LOG=warn` with validation enabled (debug builds).
- **`cmd_clear_attachments` constraint.** Spec requires it inside
  an active render pass; `vkCmdBeginRendering` must precede.
  Forgetting `loadOp=LOAD` will erase the existing window
  content. Verify by smoke-checking xterm scrollback survives a
  fill.
