# Stage 2 implementation plan — minimal-Vk correct baseline

Date: 2026-05-16
Status: draft v2 (post codex round 1), for codex + author review
Spec: `docs/superpowers/specs/2026-05-15-rendering-model-v2.md` (§ Migration → Stage 2)
Branch: `rendering-model-v2`
Predecessor: `docs/superpowers/plans/2026-05-16-stage-1.md` (landed `56ad631`, `982f1eb`)

## Goal

Stand up the **whole v2 vertical slice in Vk** with a narrow set of
paint ops, so the four-component model
(`PlatformBackend` / `DrawableStore` / `RenderEngine` /
`SceneCompositor`) is exercised end-to-end. After Stage 2 closes:

- `YSERVER_RENDER_MODEL=v2` boots and the screen displays a
  composed scanout of root + windows (no text yet).
- Fill / copy / put_image paint paths produce pixel-correct
  output against a CPU oracle.
- Damage flows storage → presentation → output via the two-list
  I5 model with snapshot-based `peek_presentation_damage` /
  `ack_presentation_damage`.
- Page-flip retirement (I6b) gates BO reuse; render-completion
  fence (I6a) gates source-storage release.
- Zero `vkQueueWaitIdle` on the steady-state hot path
  (paint + composite).
- Buffer-age partial-redraw produces frame-for-frame
  pixel-identical output to a full-redraw oracle on a synthetic
  damage pattern for ≥ 2× the scanout BO count consecutive
  frames.

What **doesn't** ship in Stage 2:

- RENDER ops, glyph atlas, text, gradients, traps, triangles —
  Stage 3.
- COMPOSITE redirect actually painting somewhere visible —
  Stage 4 (the bookkeeping side is already in from Stage 1b).
- Cursor plane (parked per I7; cursor renders via blit through
  the scene). HW cursor is Stage 5.
- Direct scanout. Disabled entirely per I7.
- Real-app acceptance — xterm/MATE/xfce/etc. need glyphs.
  Stage 3 gates real apps.
- ARGB visuals beyond BGRA8_UNORM at the scanout BO format.
  Depth-1/depth-8 pixmaps live in `DrawableStore` as storage
  (`R8_UNORM` for both depth-1 and depth-8 — depth-1's bit-
  packed protocol shape gets unpacked at `put_image` into
  byte-per-pixel `R8`; the `image_text*`-style usage that
  exercises real depth-1 masks doesn't ship until Stage 3).
- GC planemask / non-`GXcopy` raster ops in v2. v1 supports
  these via `logic_fill_pipeline`; v2 fill / copy / put_image
  ignore planemask + reject `function != GXcopy` with a logged
  gap. Comes back in Stage 3.
- Same-image / overlapping `CopyArea` (src and dst share
  storage, or the dst rect overlaps the src rect). Stage 2
  detects the overlap case and routes through a scratch image
  via copy-out / copy-in; the slow path is acceptable here
  since real apps that hit this (xterm scroll under
  non-composited WMs) are Stage 3 territory.

## Non-goals (deferred to later substages or stages)

1. **No per-method fallback to v1.** Reaffirming spec § Stage 2:
   `YSERVER_RENDER_MODEL=v2` instantiates `KmsBackendV2` as the
   *whole* `Backend` impl. Paint ops not yet implemented in v2
   log a gap and return Ok; they do not fall through to v1.

2. **No pixman, no CPU shadow buffers, no readback on the
   composition hot path.** SIGUSR1 dump_scanout remains an
   off-hot-path debug readback (spec-exempt). `get_image`
   readback is also exempt — sync-RPC by protocol design.

3. **No DRI3 / Present in v2 Stage 2.** v1 keeps DRI3 working
   for `YSERVER_RENDER_MODEL=v1`; v2's `dri3_capabilities()`
   stays `Dri3Caps::unsupported()`. DRI3 in v2 reopens after
   Stage 3 / Stage 5.

4. **No timeline-semaphore optimization.** I6a uses an
   exportable binary `VkSemaphore` (for KMS `IN_FENCE_FD`) plus
   a `VkFence` (for CPU-side lifetime polling). Stage 5 swaps
   in timeline / syncobj.

5. **No COW first-class storage.** The COW xid is still the
   `COMPOSITE_OVERLAY_WINDOW.0` placeholder from v1's
   `GetOverlayWindow`. Stage 4 makes it a real `DrawableStore`
   entry.

6. **No SHAPE-bounded blit clipping in the scene.** SHAPE
   regions accumulate in `KmsCore.shape_*` (already from
   Stage 1) but `SceneCompositor` ignores them in Stage 2.
   Stage 4 brings them back alongside redirect-state and
   projection-correctness tests called out in spec Risk 2.

7. **No hardware-plane assignment, no submit aggregation, no
   descriptor caching past the trivial level.** All strategy
   plug-ins are Stage 5.

## Substages (revised after codex round 1)

Stage 2 is too large for a single commit. **Six** substages —
one more than the v1 draft after codex's "2d is three problems
under one heading" critique. Each produces a build that passes
its own gates. Each maps to one commit on `rendering-model-v2`.
A regression in 2d, for instance, can be isolated by `git
bisect` without unwinding 2e/2f.

## Cross-cutting concepts (apply across all substages)

These were called out as load-bearing in codex round 1 and need
a unified shape before any substage starts.

### 1. Two-sync-object model: `FenceTicket` + KMS sync_file

I6a needs to gate **two** consumers with different needs. The
CPU needs to know when GPU work has completed so storage /
fence handles can be recycled. The KMS atomic commit needs a
`sync_file` fd to wait on before flipping the BO that holds the
composed image. A single `vk::Fence` doesn't fit both: fences
aren't exportable as sync_file, and recycling a fence while
some other consumer still names it is a use-after-free.

v2's answer:

```rust
// crates/yserver/src/kms/v2/platform.rs

/// A submission's CPU-side lifetime ticket. Holds a refcount
/// on a `vk::Fence` allocated from the pool; only when refcount
/// hits zero AND the fence is signaled do we recycle the fence.
/// Cloneable: every consumer (each drawable touched by the
/// submission, the SceneCompositor's pending-ack list, etc.)
/// gets a clone.
#[derive(Clone)]
pub(crate) struct FenceTicket {
    inner: Rc<FenceTicketInner>,
}

struct FenceTicketInner {
    fence: vk::Fence,
    /// Set on the first `poll_signaled` that observes
    /// VK_SUCCESS. After this, `poll_signaled` short-circuits
    /// without an actual driver call.
    signaled_cache: Cell<bool>,
    /// Returned to platform's pool on Drop iff signaled.
    /// Unsignaled-and-dropped is a renderer-failed condition
    /// (storage retirement would race the still-pending GPU
    /// work); the inner's Drop logs and leaks the fence handle.
    platform: Weak<RefCell<FencePool>>,
}

impl FenceTicket {
    pub(crate) fn poll_signaled(&self, vk: &VkContext) -> bool;
    pub(crate) fn wait(&self, vk: &VkContext) -> Result<(), vk::Result>;
}

/// A submission's KMS-side lifetime ticket. Holds an exportable
/// binary `VkSemaphore` — the compose submit signals it via a
/// signal-semaphore slot; the platform exports its sync_file
/// fd for IN_FENCE_FD on present_scanout. Distinct lifetime
/// from FenceTicket because the semaphore is consumed by the
/// kernel at flip time, not by the host.
pub(crate) struct KmsSyncSemaphore {
    semaphore: vk::Semaphore,
    fd_exported: Cell<bool>,  // exporter is one-shot per submit
}

impl KmsSyncSemaphore {
    pub(crate) fn export_sync_file(&self, vk: &VkContext)
        -> Result<OwnedFd, vk::Result>;
}
```

Single-threaded core invariant lets us use `Rc` / `RefCell`
rather than `Arc` / `Mutex` here — `KmsBackendV2` is constructed
and accessed only on the core thread.

`PlatformBackend::submit_cb` takes `signal_kms_semaphore:
Option<vk::Semaphore>`. Paint submits pass `None`; compose
submits pass `Some(semaphore_pool.acquire())`. The semaphore
handle the caller passes in is owned by the pool — after the
caller calls `export_sync_file` to extract IN_FENCE_FD, the
semaphore is dropped into the platform's "pending retirement"
list and recycled to the pool on the matching page-flip
retirement. Compose-CB callers therefore do not see a
`KmsSyncSemaphore` *struct* — they hold a raw `vk::Semaphore`
acquired from `acquire_kms_sync_semaphore`. (The `KmsSyncSemaphore`
typed wrapper in the API sketch is purely a documentation aid;
the canonical signature is `Option<vk::Semaphore>` as in
`submit_cb`. Earlier "submit returns (FenceTicket, KmsSyncSemaphore)"
wording is superseded by this.)

### 2. Image layout state machine per drawable

`DrawableStore::Drawable` carries a `current_layout:
vk::ImageLayout` field. Every op records the post-transition
layout. Before any op that needs a specific layout, an
`ImageMemoryBarrier2` transitions to it. The full state set in
Stage 2:

- `UNDEFINED` — fresh allocation, before first paint
- `TRANSFER_DST_OPTIMAL` — during put_image upload or
  copy_area dst
- `TRANSFER_SRC_OPTIMAL` — during copy_area src or get_image
  readback
- `COLOR_ATTACHMENT_OPTIMAL` — during fill_rectangle render
- `SHADER_READ_ONLY_OPTIMAL` — sampled by SceneCompositor blit

Transitions are recorded inline in the op's CB. The compose
CB begins with a per-source-drawable barrier batch that
transitions each scene entry's storage from whatever the
previous frame left it in (typically `SHADER_READ_ONLY_OPTIMAL`,
or `COLOR_ATTACHMENT_OPTIMAL` if it was painted this frame) into
`SHADER_READ_ONLY_OPTIMAL`. The corresponding access flags are
the full producer mask:

- `srcStageMask =
    COLOR_ATTACHMENT_OUTPUT | TRANSFER | (FRAGMENT_SHADER if compose-read)`
- `srcAccessMask =
    COLOR_ATTACHMENT_WRITE | TRANSFER_WRITE`
- `dstStageMask = FRAGMENT_SHADER`
- `dstAccessMask = SHADER_SAMPLED_READ`

(Codex round 1, point 2: original plan missed
`COLOR_ATTACHMENT_WRITE`. Fixed.)

A `LayoutTracker` helper on `Drawable` exposes
`record_transition_to(&mut self, cb, new_layout,
src_stage_mask, src_access_mask, dst_stage_mask,
dst_access_mask)`. Single source of truth for what the
current layout is, avoiding the "layout vs barrier flags
out of sync" class of bug.

**Write-after-read hazard** (codex round 2 supplementary): the
compose-prologue write→sampled-read mask covers paint-then-
compose-reads-the-result. The reverse direction — compose
sampled the storage in frame N, then frame N+1 paints into it
— requires the paint op's barrier `srcAccessMask` to include
`SHADER_SAMPLED_READ` (or zero-with-execution-dependency-only)
to drain the prior compose's reads. `record_transition_to`
callers in `RenderEngine` ops that target a drawable
previously sampled by compose include `SHADER_SAMPLED_READ`
in `srcAccessMask`. The validation layer catches missed cases.

### 3. Compose-after-paint ordering — same queue, in-CB barrier

Spec § Stage 2 invariant: zero `vkQueueWaitIdle` on the hot
path. The unique mechanism is **same-queue submission order +
in-CB memory barrier in the compose CB's prologue**:

- Paint submit happens first (RenderEngine's flush returns a
  FenceTicket; the underlying `vkQueueSubmit2` has been called).
- Compose submit happens immediately after on the same queue.
  Submission order on the same queue establishes execution
  order for the device-side scheduler.
- The compose CB's first commands are layout transitions on
  every source drawable that paint just wrote. Those transitions
  carry the full memory barrier (srcStageMask/Access covering
  paint's write, dstStageMask/Access covering compose's
  sampler reads).
- No CPU fence wait between paint flush and compose submit.
  The compose submit's signal fence (`FenceTicket` for I6a)
  fires when both paint and compose CBs have completed —
  which is what storage lifetime needs.

This is the path codex round 1 verified as correct in shape;
the access-flag list above resolves codex's point 2 caveat.

### 4. PaintBatch decision: do NOT reuse v1's directly

Codex round 1, point 6, is correct. v1's
`kms::scheduler::paint_batch::PaintBatch` carries v1 semantics:
`dirty_outputs: Vec<usize>` (v1's scanout-walk plumbing),
`BatchFlushReason` enum (v1's flush-trigger taxonomy with
`VisibleComposite` etc. referencing v1's compose model),
mirror-sampling comments throughout, and resource-retirement
that's tangled with v1's storage layout.

**Decision: v2 builds its own `RenderBatch` inside `RenderEngine`.**
Smaller scope; designed for v2's `FenceTicket` ownership; no
`dirty_outputs` (v2 doesn't have per-output paint batching —
paint targets are drawable storage which lives outside output
geometry). Reuses what's genuinely renderer-agnostic from v1's
sibling modules:

- `kms::scheduler::batch_upload_arena::BatchUploadArena` — pure
  Vk buffer + offset bump-allocator; no v1 dependencies.
- `kms::scheduler::batch_descriptor_arena::BatchDescriptorArena`
  — same.

These two move to `crates/yserver/src/kms/vk/batch_arenas.rs`
in 2c so v1 and v2 both depend on the shared module. The
extraction is mechanical (rename `use crate::kms::scheduler::...`
sites; no logic changes). Treated as a sibling commit landing
on top of 2b before 2c proper, or merged into 2c if the diff
size stays small. Either way it's a single mechanical move.

### 5. Compose-read consumer tracking on `FenceTicket`

Codex round 1, point 4: storing a single "latest fence ticket"
per drawable is safe **only if every consumer that reads or
writes the drawable touches the ticket**. v2 enforces this by:

- `RenderEngine::flush` produces a `FenceTicket` and calls
  `store.touch_render_fence(target_id, ticket.clone())` for
  every drawable the batch wrote.
- `SceneCompositor::tick` produces a compose `FenceTicket` and
  calls `store.touch_render_fence(source_id, ticket.clone())`
  for every drawable the compose CB sampled.
- "Touch" coalesces: if the existing ticket is already
  signaled, replace; if not, replace anyway — same-queue
  submission order guarantees the new ticket signals after the
  old. (When `wait` is called, it waits on the newest, which
  is sufficient.)

The `Rc<FenceTicketInner>` refcount ensures that even though
`Drawable.last_render_ticket` is overwritten, any other
clone (e.g. `SceneCompositor`'s pending-ack list still holds
the prior compose's ticket) keeps the fence handle alive.

## Substage 2a — `PlatformBackend` real

### Goal

Move DRM device, render-node fd, output layouts, libinput
context, scanout BOs, VkContext, command pool, fence pool out
of `KmsBackendV2`'s flat field set into a real `PlatformBackend`
struct. Expose the **two-sync-object** API
(`FenceTicket` for CPU lifetime, `KmsSyncSemaphore` for KMS
`IN_FENCE_FD`).

Behavior-preserving for v2's paint surface (still all logged
gaps), but the platform layer underneath is where v2 expects
it for 2b+.

### Module shape

```rust
// crates/yserver/src/kms/v2/platform.rs

pub(crate) struct PlatformBackend {
    // DRM / output side
    pub(crate) device: Arc<drm::Device>,
    pub(crate) render_node_fd: Option<OwnedFd>,
    pub(crate) render_node_path: Option<PathBuf>,
    pub(crate) outputs: Vec<OutputLayout>,
    pub(crate) fb_w: u16,
    pub(crate) fb_h: u16,

    // Input side
    input_ctx: Option<crate::input::SendContext>,

    // Vulkan side
    pub(crate) vk: Arc<VkContext>,
    pub(crate) ops_command_pool: vk::CommandPool,
    pub(crate) fence_pool: Rc<RefCell<FencePool>>,
    pub(crate) semaphore_pool: Rc<RefCell<SemaphorePool>>,

    // Per-output scanout BO pools (reuses v1's `ScanoutPool`).
    pub(crate) scanout_pools: Vec<ScanoutPool>,

    // I6b retirement signal infra. Per BO: last_present_generation
    // (None = freshly allocated or invalidated). Per output:
    // next_present_generation counter.
    pub(crate) bo_generations: Vec<Vec<BoGenerationEntry>>,
    pub(crate) next_present_generation: u64,

    // Lifecycle flags
    pub(crate) renderer_failed: bool,
    pub(crate) shutting_down: bool,
}

pub(crate) struct FencePool { /* recyclable VkFence stack */ }
pub(crate) struct SemaphorePool { /* recyclable exportable binary VkSemaphore stack */ }

pub(crate) struct BoGenerationEntry {
    pub(crate) last_present_generation: Option<u64>,
    /// True after a failed submit/flip where this BO's contents
    /// became indeterminate. Cleared on next successful render
    /// into this BO with full_redraw_fallback.
    pub(crate) content_invalidated: bool,
}
```

### Primitive APIs (final form per cross-cutting §1)

```rust
impl PlatformBackend {
    /// CPU-side lifetime: acquire a fresh FenceTicket whose
    /// inner VkFence is unsignaled. Returned in pre-record state
    /// so the caller can pass `ticket.inner.fence` to vkQueueSubmit2.
    pub(crate) fn acquire_fence_ticket(&mut self) -> FenceTicket;

    /// KMS-side lifetime: acquire a fresh exportable
    /// binary VkSemaphore. Used as the compose submit's signal-
    /// semaphore; export_sync_file extracts the IN_FENCE_FD.
    pub(crate) fn acquire_kms_sync_semaphore(&mut self) -> KmsSyncSemaphore;

    /// I6b: pick the next BO to render into for `output_idx`,
    /// or `None` if all BOs are still in flight.
    pub(crate) fn acquire_scanout_bo(&mut self, output_idx: usize)
        -> Option<ScanoutBoToken>;

    /// Submit a paint or compose CB.
    /// - `signal_fence` is the FenceTicket's underlying fence.
    /// - `signal_kms_semaphore` is Some only for compose submits
    ///   that will feed an atomic commit. Paint submits pass None
    ///   — paint never feeds KMS directly. Per cross-cutting §1,
    ///   the semaphore handle is one-shot: after `export_sync_file`
    ///   it is queued for pool-return on next page-flip-complete.
    pub(crate) fn submit_cb(
        &mut self,
        cb: vk::CommandBuffer,
        signal_fence: vk::Fence,
        signal_kms_semaphore: Option<vk::Semaphore>,
    ) -> Result<(), vk::Result>;

    /// Page-flip the BO. The kernel waits on the sync_file fd
    /// exported from `kms_sync` before flipping.
    pub(crate) fn present_scanout(
        &mut self,
        output_idx: usize,
        bo: ScanoutBoToken,
        kms_sync: KmsSyncSemaphore,
    ) -> Result<(), io::Error>;

    /// Page-flip-complete callback driven by the core loop on
    /// DRM fd events.
    pub(crate) fn on_page_flip_complete(&mut self, output_idx: usize)
        -> Option<PageFlipRetirement>;

    /// Invalidate a BO's content tracking — called on failed
    /// submit / failed atomic commit. Next acquire that picks
    /// this BO sees content_invalidated=true and forces full
    /// redraw.
    pub(crate) fn invalidate_bo(&mut self, output_idx: usize, bo_idx: usize);

    pub(crate) fn poll_fds(&self) -> Vec<(RawFd, BackendFdKind)>;
}

pub(crate) struct ScanoutBoToken {
    pub(crate) output_idx: usize,
    pub(crate) bo_idx: usize,
    pub(crate) image: vk::Image,
    pub(crate) image_view: vk::ImageView,
    pub(crate) extent: vk::Extent2D,
    pub(crate) last_present_generation: Option<u64>,
    pub(crate) content_invalidated: bool,
}

pub(crate) struct PageFlipRetirement {
    pub(crate) retired_bo_idx: usize,
    pub(crate) presented_bo_idx: usize,
    pub(crate) generation: u64,
}
```

The IN_FENCE_FD plumbing reuses `kms::vk::sync::export_sync_file`,
which already exports a sync_file fd from a binary VkSemaphore.
That helper is renderer-agnostic.

## Substage 2b — `DrawableStore` real

### Goal

Real storage allocation, refcount, I6a retirement gating via
`FenceTicket`, the two damage lists with snapshot/ack
semantics, image-layout state tracking, and the
`set_scene_participating` rule that clears unpresented
presentation damage.

After 2b, `Backend` methods that allocate drawables create
real `DrawableStore` entries with a `VkImage` allocated through
`PlatformBackend`. Paint paths still log gaps; entries stay
cleared (zero-filled on allocate).

```rust
        height: u16,

    pub(crate) fn get(&self, id: DrawableId) -> Option<&Drawable>;
    pub(crate) fn get_mut(&mut self, id: DrawableId) -> Option<&mut Drawable>;

    pub(crate) fn incref(&mut self, id: DrawableId);
    /// Decref; if refcount hits zero, decide whether to destroy
    /// now or queue for I6a retirement.
    pub(crate) fn decref(&mut self, platform: &mut PlatformBackend, id: DrawableId)
        -> RetireDecision;

    /// Flip scene-participation. **When set to false**:
    ///   - clear unpresented presentation_damage
    ///   - bump presentation_damage_epoch (so any in-flight
    ///     snapshot's ack becomes a no-op on the now-empty
    ///     live damage — see snapshot/ack rule below)
    /// Protocol damage is unaffected (DAMAGE-ext clients still
    /// see paint events).
    /// **When set to true**: nothing to clear; future paint
    /// accumulates.
    pub(crate) fn set_scene_participating(&mut self, id: DrawableId, v: bool);

    /// Accumulate damage. Presentation list updated only if
    /// `scene_participating` AND scene-relevant. Protocol list
    /// updated unconditionally. Bumps presentation_damage_epoch
    /// iff presentation_damage was updated.
    pub(crate) fn damage(&mut self, id: DrawableId, rect: vk::Rect2D);

    /// Snapshot for composite. Caller passes back to ack.
    pub(crate) fn peek_presentation_damage(&self, id: DrawableId)
        -> DamageSnapshot;

    /// Ack rule (codex round 1, restated): if snap.epoch ==
    /// current_epoch, clear live damage. If snap.epoch <
    /// current_epoch, subtract only the snapshot's region from
    /// live damage. (Paint between peek and ack survives.)
    /// **In addition** (codex round 1, point 5): if
    /// scene_participating was flipped to false since the
    /// snapshot was taken, this is a no-op (presentation_damage
    /// was cleared by set_scene_participating(false); subtracting
    /// the snapshot's region from an empty set is harmless but
    /// we still don't bump the epoch).
    pub(crate) fn ack_presentation_damage(
        &mut self,
        id: DrawableId,
        snap: DamageSnapshot,
    );

    pub(crate) fn peek_protocol_damage(&self, id: DrawableId) -> RegionSet;
    pub(crate) fn subtract_protocol_damage(
        &mut self,
        id: DrawableId,
        region: &RegionSet,
    );

    /// Record an in-flight ticket. Coalesces: any prior ticket
    /// is dropped (its Rc inner remains alive in other holders
    /// like SceneCompositor's pending-ack list). Per
    /// cross-cutting §5, EVERY consumer that reads OR writes the
    /// drawable must call this — RenderEngine for writes,
    /// SceneCompositor for reads.
    pub(crate) fn touch_render_fence(
        &mut self,
        id: DrawableId,
        ticket: FenceTicket,

    pub(crate) fn record_layout_transition(

        src_access: vk::AccessFlags2,
        dst_stage: vk::PipelineStageFlags2,
        dst_access: vk::AccessFlags2,
    );

    /// last_render_ticket is signaled (or None), destroy
    /// Vk handles via platform and remove from the map.
```

## Substage 2c — `RenderEngine`: fill / put_image / get_image

### Goal

Three paint ops cross the v2/Vk gap and mutate storage in
`DrawableStore` via v2's own `RenderBatch` (NOT v1's
`PaintBatch`). After 2c, calling `Backend::fill_rectangle` /
`put_image` on `KmsBackendV2` actually writes pixels into the
target `Drawable`; `Backend::get_image` reads them back via
staging. `copy_area` is deferred to 2d (its overlapping-region
scratch path is non-trivial enough to deserve its own slot,
per codex round 1 point 8). SIGUSR1 `dump_scanout` does **not**
yet show paint output (no scene compose yet — 2d).

### Ops

| Backend method | v2 op | Vk impl |
|---|---|---|
| `fill_rectangle` | `engine.fill_rect(target, rect, color)` | Fullscreen-quad-with-scissor pipeline → records `vkCmdDraw` after layout barrier into `COLOR_ATTACHMENT_OPTIMAL`. GC `function != GXcopy` or non-zero planemask logs a gap + returns Ok. |
| `put_image` | `engine.put_image(target, dst_pos, bytes, format)` | Staging buffer via `BatchUploadArena` + `vkCmdCopyBufferToImage`. Layout transition `_ → TRANSFER_DST_OPTIMAL`. Depth-1 client data is unpacked to R8 before staging. |
| `get_image` | `engine.get_image(src, rect)` | `vkCmdCopyImageToBuffer` to a readback staging buffer. Synchronous: submits, waits on the FenceTicket via `wait`, copies bytes out. This is the **only** sync path. |

Other paint methods (`copy_plane`, `poly_*`, `image_text*`,
`fill_poly`, all `render_*`) stay logged-gap stubs through 2c.
`copy_area` joins them — its impl lands in 2d.

### Where 2c's submit boundary is

Codex round 1 point 1: 2c's acceptance is **offscreen-only**
because no scene exists to make paint visible. `get_image`
is the test oracle. The hot-path sync rule from cross-cutting
§3 isn't exercised at 2c (no compose yet) but the
infrastructure (FenceTicket, layout tracker, touch_render_fence)
is in place.

Every `fill_rect` / `put_image` call:
1. If batch not open, open one (allocate CB, begin recording).
2. Record layout transition on target into the op's required
   layout.
3. Record the op.
4. Push `target` onto `batch.touched_drawables` (deduplicated).
5. Call `store.damage(target, rect)`.

`flush` then submits, ticket is recorded on every touched
drawable.

`get_image`:
1. Force-flush any open batch (its ticket records on
   touched drawables).
2. Open a one-shot CB. Layout transition src to
   `TRANSFER_SRC_OPTIMAL`. `vkCmdCopyImageToBuffer`.
3. Submit; wait on the resulting ticket.
4. Map readback buffer, copy bytes out, unmap.
5. Return bytes.

## Substage 2d — `RenderEngine::copy_area` + scene graph + damage projection (no buffer-age yet)

### Goal

Two things land together because copy_area needs DrawableStore
mature (which 2b delivered) but the scene's first-visible
output needs at minimum a blit pipeline that samples from
DrawableStore. Building scene-graph + projection + blit in the
same commit avoids a half-built scene that can't render
anything.

**No buffer-age clipping yet**: every tick is a full-output
redraw. This makes 2d testable in isolation without the
buffer-age algorithm's correctness gating (which is 2e). Per
codex round 1 point 8, splitting "scene + projection" from
"buffer-age + retirement integration" is the clean cut.

### `copy_area`

| Subcase | Vk path |
|---|---|
| `src != dst` (any rect) | Layout transitions: src → `TRANSFER_SRC_OPTIMAL`, dst → `TRANSFER_DST_OPTIMAL`. `vkCmdCopyImage` with the requested rect / offset. |
| `src == dst` (any rect, overlap or not) | **Always scratch path** (codex round 2 point 3). A single `VkImage` cannot be in both `TRANSFER_SRC_OPTIMAL` and `TRANSFER_DST_OPTIMAL` simultaneously for the same subresource — using `GENERAL` for both layouts would technically validate, but the spec discourages it and `GENERAL` performance varies across vendors. Allocate a temporary `vk::Image` from the upload arena's image-side at the rect's size, layout `TRANSFER_DST_OPTIMAL`. Source image → `TRANSFER_SRC_OPTIMAL`; copy src rect → scratch; barrier on source image back to its prior layout (or, after the second copy completes, directly to `TRANSFER_DST_OPTIMAL`); copy scratch → dst rect. Scratch's destruction adopted by the batch (retires on FenceTicket signal). The slow path is acceptable in Stage 2 because real apps that hit this (xterm scroll without compositor) need glyphs to be relevant anyway, and they land in Stage 3. |

### Compose tick (no buffer-age — 2d shape)

1. Rebuild scene from `core.top_level_order` +
   `windows_v2[xid]` geometry + cursor position. Order from
   spec: root → top-levels descend (Manual-redirected skipped
   per I4) → cursor. **Multi-output**: each entry is clipped to
   each output's geometry and appears in that output's tick.
   (Multi-output flip ordering is a Risk for 2e.)
2. For each scene entry, call `peek_presentation_damage` and
   record the snapshot for ack. **2d doesn't use the damage
   region for clipping** — every tick repaints the full output —
   but the snapshot machinery flows so 2e can wire it without a
   schema change.
3. Acquire BO from `platform`. Acquire FenceTicket and
   KmsSyncSemaphore.
4. Open compose CB:
   - For each scene-entry source storage: barrier from current
     layout → `SHADER_READ_ONLY_OPTIMAL` (per cross-cutting §2
     full producer mask).
   - BO layout `UNDEFINED → COLOR_ATTACHMENT_OPTIMAL`.
   - `vkCmdBeginRendering` with BO as color attachment.
   - Optional `loadOp=CLEAR` clear (Stage 2d uses CLEAR every
     frame since we're full-redrawing).
   - For each scene entry in z-order: bind blit pipeline +
     per-entry descriptor (sampler + storage's image_view), draw
     fullscreen quad scissored to entry.bounds with UV from
     entry.source.
   - End rendering. BO layout `→ PRESENT_SRC_KHR`.
5. `platform.submit_cb(cb, ticket.fence, Some(kms_semaphore))`.
6. For each scene-entry source: `store.touch_render_fence(id,
   ticket.clone())`.
7. `platform.present_scanout(output_idx, bo, kms_semaphore)`.
8. Stash the recorded snapshots in `pending_acks` keyed by
   `output_idx + bo_idx + ticket_generation` for later ack on
   page-flip-complete.
9. On page-flip-complete (`on_page_flip_ready`):
   - `platform.on_page_flip_complete(output_idx)` → retirement
     info.
   - For each pending_ack matching the retired flip: call
     `store.ack_presentation_damage(id, snap)`.
   - `engine.poll_retired(vk)`, `store.poll_pending_retire(platform)`.
10. On failed submit / failed flip:
    - `platform.invalidate_bo(output_idx, bo_idx)`.
    - Do not ack damage.
    - Drop the FenceTicket the compose submit would have
      signaled — but only if the underlying fence never got into
      the submit. If submit succeeded but the flip failed, the
      ticket DOES signal eventually; we still drop our clone
      because the ack path bails. Storage refs continue to hold
      their own clone.
    - Bump `scene_structure_dirty` so the next tick re-renders.

## Substage 2e — Buffer-age clipping + I6b retirement + failed-flip recovery

### Goal

Layer buffer-age clipping + the snapshot/ack plumbing on top
of 2d's full-redraw scene. After 2e, partial redraws are
correct under all the cases codex round 1 raised:

- buffer-age algorithm at BO pick time uses
  `current_output_damage ∪ history(bo.last_present_gen,
  current_gen)` — the current frame's damage is not yet in
  the history when this picks.
- BO content invalidated on failed submit/flip → next pick
  forces full redraw.
- Snapshot/ack mid-frame paint survival.

### Types (additions to 2d)

```rust
pub(crate) struct OutputSceneState {
    pub(crate) output_idx: usize,
    pub(crate) extent: vk::Extent2D,
    pub(crate) damage_history: BufferAgeRing,
    pub(crate) current_generation: u64,
    pub(crate) scene_structure_damage: RegionSet,  // output coords
    pub(crate) pending_repaint_after_failed_submit: RegionSet,
    pub(crate) pending_acks: VecDeque<PendingAck>,
}

/// Ring depth = max(scanout_bo_count) + 1. The +1 buys edge
/// safety (codex round 1 point 3 conservative tip).
pub(crate) struct BufferAgeRing { entries: VecDeque<(u64, RegionSet)>, depth: usize }

pub(crate) struct PendingAck {
    pub(crate) generation: u64,
    pub(crate) bo_idx: usize,
    /// Per-drawable damage snapshots taken at tick entry — acked
    /// per drawable on retirement.
    pub(crate) drawable_snapshots: Vec<DamageSnapshot>,
    /// **Output-level damage snapshot** (codex round 2 point 1):
    /// the exact regions folded into this frame's compose. On
    /// retirement, the platform subtracts ONLY these regions
    /// from live `scene_structure_damage` and
    /// `pending_repaint_after_failed_submit`. Damage that arrived
    /// between submit and retirement (map/unmap/cursor-move while
    /// the frame was in flight) is NOT in this snapshot and
    /// therefore survives, driving the next tick.
    pub(crate) submitted_output_damage: RegionSet,
    pub(crate) submitted_scene_structure_damage: RegionSet,
    pub(crate) submitted_failed_repaint: RegionSet,
    pub(crate) ticket: FenceTicket,
}
```

### Buffer-age BO pick (codex round 1 point 3 corrected)

```text
fn pick_repaint_region(output, bo):
    frame_gen = output.current_generation + 1
    current_damage = output.compute_current_output_damage()   // from peek + scene-structure
    
    if bo.content_invalidated or bo.last_present_generation is None:
        return Full(extent)
    
    last_gen = bo.last_present_generation
    if last_gen >= frame_gen:
        // Shouldn't happen; bail safe.
        return Full(extent)
    
    history_gens = (last_gen+1, last_gen+2, ..., frame_gen-1)
    if not output.damage_history.contains_all(history_gens):
        return Full(extent)  // history loss → fallback
    
    repaint = current_damage
    for g in history_gens:
        repaint = union(repaint, output.damage_history[g])
    return Clipped(repaint)
```

### Tick algorithm (final, replacing 2d's full-redraw)

1. `current_output_damage` = union of:
   - projected `peek_presentation_damage` for each scene entry
   - `scene_structure_damage`
   - `pending_repaint_after_failed_submit`
2. If `current_output_damage` is empty → skip everything.
3. Pick next BO. If `acquire_scanout_bo` returns `None`, skip
   this tick — next core-loop iteration retries.
4. `repaint_region = pick_repaint_region(output, bo_token)`.
   If `Clipped(empty)` after intersect with output extent →
   skip.
5. Build compose CB (as in 2d) but **scissor scene-entry
   draws to `(entry.bounds ∩ repaint_region)`**. Source UV
   recomputes from the scissored output rect.
6. Submit the compose CB:
   - `platform.submit_cb(cb, ticket.fence, Some(kms_sem))`.
     If this returns Err → go to step 9a (queue-submit-failed
     path).
6a. **Touch compose-source drawables with the ticket**
    (cross-cutting §5; explicit step for clarity per codex
    round 3 finding 1): for each scene-entry source storage,
    call `store.touch_render_fence(source_id, ticket.clone())`.
    This is what keeps the compose ticket's inner `Rc` alive
    via storage's `last_render_ticket` independent of whether
    present_scanout succeeds — fence-lifecycle correctness for
    branch 9b depends on this.
7. Present:
   - `platform.present_scanout(output_idx, bo, kms_sem)`. If
     atomic commit returns Err → go to step 9b (atomic-commit-
     failed path).
8. **Bookkeeping at flip-pending boundary** (codex round 2
   point 2 — transactional commit; advancement happens ONLY
   after both the queue submit and the atomic commit have
   accepted). Both `submit_cb` and `present_scanout` returned
   Ok, so a flip is pending:
   - Build `PendingAck` with `submitted_output_damage =
     current_output_damage`, `submitted_scene_structure_damage
     = scene_structure_damage.clone()`,
     `submitted_failed_repaint =
     pending_repaint_after_failed_submit.clone()`,
     `drawable_snapshots = pending_snapshots_from_tick_entry`,
     `ticket`, `generation = frame_gen`, `bo_idx`.
   - Push onto `output.pending_acks`.
   - Advance `output.current_generation = frame_gen`.
   - **Do NOT** clear `scene_structure_damage` or
     `pending_repaint_after_failed_submit` yet — the snapshot
     captures what *this* frame covers; retirement does the
     subtraction. New damage that arrives after this point but
     before retirement stays in the live regions and will be
     picked up by the next tick (or, if it lands before
     retirement, by the retirement's *subtract-only-snapshot*
     rule).
9. On retirement (page-flip-complete fires for this BO):
   - Find the matching `PendingAck` (front of queue should
     match `bo_idx + generation`).
   - For each `DamageSnapshot` in `drawable_snapshots`:
     `store.ack_presentation_damage(id, snap)`.
   - `scene_structure_damage = subtract(scene_structure_damage,
     pending.submitted_scene_structure_damage)`. Anything that
     arrived since submit is preserved.
   - `pending_repaint_after_failed_submit = subtract(
     pending_repaint_after_failed_submit,
     pending.submitted_failed_repaint)`.
   - Push `(frame_gen, pending.submitted_output_damage)` onto
     `output.damage_history`. Use the captured snapshot, not
     the live `current_output_damage` (which may already have
     drifted).
   - Record BO's new `last_present_generation = frame_gen`.
   - Clear `bo.content_invalidated` if previously set.
   - `engine.poll_retired(vk)`,
     `store.poll_pending_retire(platform)`.
9a. **Queue-submit-failed path** (the compose `submit_cb`
    returned Err):
    - The compose CB never reached the GPU. No BO content
      change. Do NOT call `present_scanout` or
      `invalidate_bo`.
    - Drop the FenceTicket (its inner fence was never queued
      — recycle straight to the pool; signaled_cache stays
      false but the fence wasn't submitted, so it's safe to
      reset).
    - `output.pending_repaint_after_failed_submit ∪= repaint_region`.
    - Do NOT push to `pending_acks`. Do NOT advance
      `current_generation`.
    - Mark `output.scene_structure_dirty` so the next tick
      retries.
9b. **Atomic-commit-failed path** (compose `submit_cb`
    returned Ok but `present_scanout` returned Err — the GPU
    has rendered into the BO but KMS rejected the flip):
    - `platform.invalidate_bo(output_idx, bo_idx)` — BO
      contents are now indeterminate (rendered-but-not-shown,
      potentially racy across vt-switch).
    - Drop the KmsSyncSemaphore (it's signaled by the queue
      submit; export-sync-file may have been called or not,
      but either way the semaphore goes back to the pool on
      next retirement via the platform's pending-retirement
      list).
    - Drop the FenceTicket clone we held for `PendingAck` —
      but only our local clone; the inner Rc keeps the fence
      alive for the storage's `last_render_ticket` until
      retirement.
    - `output.pending_repaint_after_failed_submit ∪= repaint_region`.
    - Do NOT push to `pending_acks`. Do NOT advance
      `current_generation`.
    - The BO sits idle until next acquire (which sees
      `content_invalidated=true` and full-redraws).

  frame 3 eventually signals (queue work completed); storage

  into `pending_repaint_after_failed_submit`; frame 4 picks
  a fresh BO and renders normally (no forced full-redraw on
  the surviving BO from frame 2).
- **Mid-frame paint survival**: peek damage on frame N; paint
  more damage between peek and present; ack on frame N+1
  retirement leaves the post-peek damage live; frame N+1's
  pick sees it.
- **Output-level damage survival (codex round 2 point 1)**:
  during a frame's in-flight window (between submit and
  retirement), trigger an unmap or cursor-move that lands new
  scene_structure_damage in output coords. On retirement,
  assert the PendingAck's `submitted_scene_structure_damage`
  is subtracted from live state but the post-submit region
  survives. Next tick sees it.
- **Transactional generation advance (codex round 2 point 2)**:
  three failure modes:
  - queue submit fails → no pending_ack, no generation advance,
    BO not invalidated (was never written), repaint region
    folds into pending_repaint_after_failed_submit
  - atomic commit fails after queue submit succeeded → BO
    invalidated, no pending_ack, no generation advance, repaint
    region folds into pending_repaint_after_failed_submit
  - both succeed → pending_ack pushed, generation advanced,
    retirement subtracts cleanly
- **History loss fallback**: pause for ring-depth+1 frames
  (no input, no damage) then drive a frame; ring should
  contain entries; subsequent picks shouldn't hit fallback.
  (Edge case — history is not actually lost in steady state
  with no input; this test exercises the "ring expiration
  through aging" path.)

## Risks (expanded post codex round 1)

**Risk 1 — Vk pipeline boilerplate scope creep.** Stage 2 needs
a `FillPipeline` (one fragment-shader path) and a
`BlitPipeline` (one sampler + fullscreen quad). Do not copy
v1's pipeline-builder + descriptor-pool-management infra
wholesale; build the two pipelines as direct ash calls. Refactor
when Stage 3 brings the second wave.

**Risk 2 — RenderBatch / PaintBatch divergence.** v2 builds its
own `RenderBatch` (not v1's `PaintBatch`) per codex round 1
point 6. The shared substrate is the two arena types only.
Risk: v2's RenderBatch drifts in shape from v1's PaintBatch
and divergent bugs creep in. **Mitigation:** treat the v1
`paint_batch.rs` as a reference doc when writing v2's
`render_batch.rs`; explicit comments where v2 deliberately
departs (no `dirty_outputs`, no `BatchFlushReason`, no
mirror-sampling assumptions).

**Risk 3 — Window-tree state in v2.** Stage 1 left
`windows: HashMap<u32, WindowState>` in `KmsBackend`. v2 needs
window geometry for scene assembly. **Mitigation:** minimal
parallel `windows_v2: HashMap<u32, WindowGeometryV2>` on
`KmsBackendV2` (NOT `KmsCore` — v1 doesn't need it). Stage 4
collapses it into `KmsCore.windows` when `WindowState` splits.

**Risk 4 — Root storage allocation timing.** v1 allocates root
on-demand; v2 allocates root at `PlatformBackend::open` (so
scene can always blit from a real source). Multi-output hotplug
changes the union extent. **Mitigation:** 2a allocates root at
initial fb_w/fb_h. Hotplug-reallocation is a follow-up if
multi-output is exercised; xfce-on-bee is single-output, so
this doesn't gate Stage 2.

**Risk 5 — Lavapipe vs hardware divergence on Vk layout
transitions.** Some image-layout bugs only fail on real driver
validation. **Mitigation:** enable
`VK_LAYER_KHRONOS_validation` in the test runner under
lavapipe + run a manual validation pass on bee at 2d + 2e
close. Validation errors at 2f hardware smoke are the gate.

**Risk 6 — `BatchUploadArena` lifetime under shared use.**
The extracted shared module (`kms::vk::batch_arenas`) is used
by v1 and v2 concurrently (though at session level only one
backend runs). **Mitigation:** the module owns no static
state; per-instance arena lifetime is owned by each backend's
batch. Cross-version interference impossible at session
granularity.

**Risk 7 — Page-flip-complete dispatch in v2.** `Backend`
trait's `on_page_flip_ready` is already wired by the core
loop; v2's body calls `scene.handle_flip_complete +
store.poll_pending_retire + engine.poll_retired`. **Mitigation:**
no new infra; just routing.

**Risk 8 — Hardware smoke blocked by missing text rendering.**
2f's smoke target is "cold-start + root + cursor + no faults"
— explicitly NOT "xfce session fully rendering." Documented
in the 2f commit message + the user-side smoke expectation.

**Risk 9 — `vk_queue_wait_idle = 0` in steady state.** Beyond
`get_image`, the candidate sources are: scanout pool init
(one-shot, not counted), failed-submit fallback (rare), shutdown
teardown (not steady state). 2c/2d/2e must not introduce
queue-wait-idle for ordering composition after paint — the
in-CB memory barrier handles ordering. **If 2d/2e tests show
queue-wait-idle on the hot path, treat it as a substage bug,
not a deferred issue.**

**Risk 10 — Single-threaded core invariant.** Stage 1 preserved
it. Stage 2 must not break it: `RenderEngine`,
`DrawableStore`, `SceneCompositor`, `PlatformBackend` are
all `!Send + !Sync` by construction. Backend: Send is
satisfied at the `Arc<Mutex<dyn Backend>>` boundary as in
v1. No new threads, no new locks.

### New risks added post codex round 1

**Risk 11 — Image layout tracking correctness.** Every drawable
carries `current_layout`. Every op consults + updates it. A
bug here (e.g. forgetting to update after a non-explicit
transition like `vkCmdBeginRendering`'s implicit one) corrupts
subsequent ops with a wrong barrier. **Mitigation:** centralise
all transitions through `Drawable::record_layout_transition`;
panic-in-debug-assert on transition from a state the op
doesn't expect; validation layer catches mismatches.

**Risk 12 — Fence-handle recycling vs DrawableStore references.**
`FenceTicket` uses `Rc<FenceTicketInner>` so the underlying
`vk::Fence` survives until every consumer drops its clone.
The recycle-on-Drop only fires for signaled fences; unsignaled
+ dropped leaks the fence (logged + a renderer-failed condition
flag set). **Mitigation:** the Rc + signaled-cache enforces
this in the type system. Test: drop a never-signaled ticket
without calling `wait` — leaks the fence + logs once + sets
`renderer_failed`.

**Risk 13 — Compose-read consumer tracking on FenceTicket.**
Per codex round 1 point 4: `SceneCompositor::tick` must call
`store.touch_render_fence(source_id, compose_ticket.clone())`
for every scene-entry source storage. Otherwise compose-read
in flight while paint overwrites the drawable on the next
frame loses track of the compose reader. **Mitigation:** the
tick algorithm explicitly does this in step 6 (see 2d). Unit
test: compose reads drawable D; before D's compose-ticket
signals, paint writes D again; assert D's
`last_render_ticket` chain stays correct (the new paint
ticket replaces the compose ticket, but the compose ticket's
Rc is still alive via `pending_acks`, so the underlying fence
isn't recycled prematurely).

**Risk 14 — BO content invalidation after failed flip.** Per
codex round 1 point 3 closing note: a BO whose compose
submit succeeded but whose atomic commit failed (rare but
possible — EBUSY on commit, content racy across vt-switch,
etc.) has been rendered into but holds undefined contents.
The platform's `invalidate_bo` flags this; the next pick
forces full redraw. **Mitigation:** integration test that
forces an EBUSY-style atomic commit failure (mock the DRM
ioctl in the test fixture) and verifies the invalidation
fires + the next frame full-redraws.

**Risk 15 — Depth-1 and depth-8 pixmap semantics.** Stage 2
ships R8_UNORM storage for both; depth-1 protocol data is
unpacked at `put_image`. Real depth-1 client usage is glyph
masks via the RENDER `image_text*` path, which is Stage 3.
Stage 2 only exercises depth-1 via the unit-test round-trip.
**Mitigation:** the round-trip test asserts bit-identical
pack-unpack; the format mismatch between depth-1 storage
(R8) and the scene's BGRA scanout is irrelevant at Stage 2
because no scene entry of depth-1 is in the scene
(non-window-non-cursor drawables aren't scene-participating).

**Risk 16 — GC clipping / planemask / raster ops / subwindow-
mode / SHAPE.** Stage 2 ignores all of these (logged gap on
non-default) except clip rectangles, which are honoured as
a fragment-shader uniform on `FillPipeline` (or, equivalently,
multiple per-rect `vkCmdSetScissor` + draws). Comes back in
Stage 3 alongside RENDER. **Mitigation:** documented in
"What doesn't ship in Stage 2".

**Risk 17 — Same-image overlapping CopyArea.** Stage 2d's
scratch path handles this. **Mitigation:** unit test covers
the overlap case; scratch image is adopted by the RenderBatch
and retires with the FenceTicket.

**Risk 18 — Root resize / hotplug.** Documented in Risk 4.
**Mitigation:** Stage 2 single-output only at the hardware
smoke level; hotplug surfaces are deferred to a small follow-up
or absorbed into Stage 4/5 when multi-output gets real
testing.

**Risk 19 — Descriptor / image-view lifetime under in-flight
compose.** Each scene-entry descriptor uses a per-frame
allocation from `BatchDescriptorArena`. The arena is owned
by the compose-tick's `RenderBatch`-equivalent; it retires
when the tick's FenceTicket signals. **Mitigation:** scene's
descriptor allocations live in a per-tick arena keyed to the
compose `FenceTicket`'s lifetime. Test: drop a compose tick
mid-flight (force renderer_failed); assert no descriptor-
arena leak (telemetry's `descriptor_allocations/sec` doesn't
grow unbounded).

**Risk 20 — Multi-output flip ordering.** Different outputs
flip at different times; ack lifecycles can interleave. Stage 2
hardware smoke is single-output, so this risk doesn't gate
2f. **Mitigation:** `PendingAck` is keyed by `output_idx +
