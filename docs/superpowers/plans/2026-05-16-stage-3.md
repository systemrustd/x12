## Goal

Bring v2 to **real-app parity**: every RENDER op + glyph/text op that
GTK, Cairo, xterm, and rxvt actually use lands on the Vk substrate
that Stage 2 built, with damage / I6a / I6b semantics unchanged.
After Stage 3 closes:

- `YSERVER_RENDER_MODEL=v2` runs xterm, xeyes, xclock, gedit, MATE
  control-centre, and xfce4 (no compositor) and the output is
  visually identical to v1 at the per-window / per-glyph level.
- rendercheck passes the same subset v1 passes (see "Acceptance
  gates", §"Correctness").
- bee + 30-minute window-drag + gedit-page-down + adapta-mate-cc
  session produces zero `amdgpu PERMISSION_FAULTS` and zero
  `vk_queue_wait_idle` calls in steady state (per spec § "Stability
  gate").
- Per-workload perf gates (see §"Perf") pass against captured v1
  baselines.

What v2 still cannot do after Stage 3 (deferred to Stage 4):

- COMPOSITE redirect actually paints somewhere visible (xfwm4
  with compositing, picom, xcompmgr).
- Composite Overlay Window first-class storage.
- xfce drop-shadow / picom blur.

Stage 3 is the **first stage gated on real apps**. Stage 2's
minimal-Vk substrate (fill / copy / put_image / get_image / blit)
is unchanged. Stage 3 only adds paint primitives on top of it
and the picture / glyphset / font protocol records.

## Non-goals (deferred to later substages or stages)

1. **No COMPOSITE redirect activation.** Picture records for
   redirected backings allocate as before (Stage 1b + 2b path);
   the Manual-mode "scene removes the window" rule does NOT
   come back in Stage 3 — the scene-participation flip still
   happens at map/unmap, and redirect state in `KmsCore` is
   bookkept but does not change scene composition. Stage 4 wires
   it.
2. **No per-method fallback to v1.** Same rule as Stage 2:
   `YSERVER_RENDER_MODEL=v2` instantiates `KmsBackendV2` as the
   whole `Backend` impl. Ops that v2 can't yet record (e.g. a
   PictOp we explicitly defer) log a gap and return Ok — they
   do not fall through to v1.
3. **No new strategy plug-ins.** Stage 5 owns hardware cursor
   plane, direct scanout, hardware plane assignment, and submit
   aggregation. Stage 3's perf budget is "no worse than v1 on
   the workloads v1 handles well"; the headroom wins land later.
4. **No timeline-semaphore / syncobj submission.** Same I6a
   binary `vk::Fence` + `KmsSyncSemaphore` pair from Stage 2.
   Stage 5 swaps in timeline.
5. **No descriptor-set caching past the basic level.** Each
   composite tick still rebuilds compose-side descriptor sets
   per scene entry (Stage 2 behaviour). RENDER's per-call
   descriptor sets are allocated from a per-batch
   `BatchDescriptorArena` (carried over from v1 — see § "Cross-
   cutting"). Persistent per-drawable view caching arrives in
   Stage 5.
6. **No glyph-atlas grow / LRU.** Same fixed 4096×4096 R8 atlas
   v1 has. Atlas-full → **drop the glyph**: paint is skipped for
   that glyph, the pen still advances by the glyph's
   `character_width`, and the engine logs `atlas_full` once per
   session + increments a `glyphs_dropped_atlas_full` counter.
   This deliberately does **not** reintroduce pixman (per spec §
   "Out of scope"); the pixman-CPU fallback v1 used here is
   gone. Codex round 1 confirmed this is the right call given
   that atlas-full has never been observed in practice (4096²
   R8 holds ~14k typical glyphs). The spec identifies the v1-era
   per-glyph `queue_wait_idle` in `GlyphAtlas::intern` as Stage-3
   fix territory; that's in scope for substage 3a (see §
   "Cross-cutting" §3).
7. **No SHAPE-bounded RENDER clipping in the scene-blit step.**
   Stage 4. RENDER paint *into* a SHAPE-bounded window's storage
   is fine (RenderEngine doesn't know about SHAPE); the scene
   ignoring SHAPE on output is the deferred bit.
8. **No `set_filter` beyond `nearest` (default).** Bilinear /
   convolution defer to Stage 5 perf work (the v1 implementation
   was abandoned on `render-convolution-filter`; the v2 attempt
   reuses the working nearest-only path).
9. **No `alpha_map` PictAttribute.** Same deferral as v1.
10. **Direct scanout, HW cursor, HW plane.** Same I7 hold.

## Cross-cutting concepts

### 1. RenderEngine grows pipelines + scratch images

Stage 2c's `RenderEngine` owns `SubmittedOp` queue + `BatchUploadArena`
+ command pool. Stage 3 promotes it to a real renderer with the
same shape v1's `KmsBackend` carried, but living on the v2
substrate:

```rust
// crates/yserver/src/kms/v2/engine.rs

pub(crate) struct RenderEngine {
    // From Stage 2 (unchanged):
    vk: Arc<VkContext>,
    command_pool: vk::CommandPool,
    submitted: VecDeque<SubmittedOp>,
    upload_arena: BatchUploadArena,

    // Stage 3a: glyph atlas + text pipeline.
    glyph_atlas: Option<GlyphAtlas>,
    text_pipeline: Option<TextPipeline>,

    // Stage 3b: pipeline + scratch.
    render_pipelines: Option<RenderPipelineCache>,
    descriptor_arena: BatchDescriptorArena,
    solid_src_image: Option<DrawableImage>,      // 1×1 R32G32B32A32_SFLOAT
    solid_mask_image: Option<DrawableImage>,     // 1×1 R32G32B32A32_SFLOAT
    white_mask_image: Option<DrawableImage>,     // 1×1 R8_UNORM == 0xFF
    dst_readback: Option<DstReadback>,            // for Disjoint/Conjoint
    /// Per-drawable sampler view cache. Keyed by DrawableId.
    /// Image view lives on Drawable but the (drawable, sampler-config)
    /// descriptor binding the RENDER pipeline expects is cached here.
    /// Per-(drawable, sampler-config, swizzle-class) sampler-
    /// binding cache (codex round 2). Key is NOT bare DrawableId
    /// because the same drawable can be sampled with different
    /// sampler / swizzle configurations (source vs mask vs
    /// alpha-only). See § Risk 15 + 3b sampler-config
    /// classification.
    drawable_view_cache:
        HashMap<(DrawableId, SamplerConfig, SwizzleClass), CachedDrawableView>,

    // Stage 3e: trap + triangle pipeline.
    trap_pipeline: Option<TrapPipeline>,

    // Stage 3f: GXcopy/planemask logic-fill pipeline.
    logic_fill_pipeline: Option<LogicFillPipeline>,

    // Stage 3 add-on to FenceTicket discipline: the
    // BatchDescriptorArena's per-batch descriptor pool is owned
    // by the SubmittedOp until the op's FenceTicket signals
    // (descriptor sets are read by the GPU; we cannot reset
    // their pool while the CB is in flight).
}
```

These adopt their **v1 implementations wholesale**: `GlyphAtlas`,
`TextPipeline`, `RenderPipelineCache`, `record_render_composite`,
`record_text_run`, `record_render_trapezoids`, `LogicFillPipeline`,
`DstReadback`, `gradient::GradientPicture`. The recorder functions
in `crates/yserver/src/kms/vk/ops/{render,text,traps}.rs` operate
on a `&mut DrawableImage`, which is v1's mirror type — v2 builds
a thin adapter that hands the recorder a `DrawableImage` view of
a `Drawable`'s storage (extent + image + image_view + layout
tracker). The recorder doesn't know whether the underlying
storage came from a v1 `WindowState.vk_mirror` or a v2
`Drawable.storage`; the adapter is purely a shape match.

**Adapter shape (concrete):** v1's `DrawableImage` is `{ vk_image,
vk_image_view, format, extent, current_layout }`. v2's `Drawable`
carries the same five fields plus damage / refcount / ticket. The
adapter is a borrow:

```rust
// crates/yserver/src/kms/v2/engine.rs
fn as_drawable_image<'a>(d: &'a mut Drawable) -> DrawableImageRef<'a> { ... }

// A &mut alias over the five fields the recorder needs. Drop
// reflects current_layout back into Drawable.
pub(crate) struct DrawableImageRef<'a> { ... }
```

This is mechanical glue. No logic changes to the recorders.

### 2. Picture record split — `KmsCore` vs `RenderEngine`

Per spec § "KmsCore scope", the picture **record** (op, src/dst
xid, transform, filter name, clip region, repeat, alpha-map xid)
lives in `KmsCore`. The GPU sampler / image view / pipeline cache
lives in `RenderEngine`.

```rust
// crates/yserver/src/kms/core.rs (Stage 3b adds:)

pub(crate) struct KmsCore {
    // ... existing fields ...
    pub(crate) pictures: HashMap<u32, PictureRecord>,
}

/// Picture record. Same shape as v1's PictureState but without
/// the GPU-state variant — gradient/solid-fill GPU state lives
/// in RenderEngine, keyed by picture_xid.
pub(crate) enum PictureRecord {
    Drawable {
        host_xid: u32,           // backing window/pixmap xid
        clip: Option<Vec<Rectangle16>>,
        clip_x: i16,
        clip_y: i16,
        repeat: Repeat,
        alpha_map: Option<u32>,
        alpha_x: i16,
        alpha_y: i16,
        component_alpha: bool,
        transform: Option<PictTransform>,
        filter: PictureFilter,   // Nearest in Stage 3; Bilinear/Conv Stage 5
        graphics_exposure: bool,
        subwindow_mode: u8,
        poly_edge: u8,
        poly_mode: u8,
    },
    SolidFill { premul: [f32; 4], repeat: Repeat, component_alpha: bool },
    LinearGradient { /* control points, stops */ },
    RadialGradient { /* control points, stops */ },
}
```

`RenderEngine` keeps a parallel `HashMap<u32, PicturePaintState>`
for the GPU-side resources of pictures that carry them
(gradients, the solid-fill's per-call uniform). The v1 split
`pictures: HashMap<u32, PictureState>` + `picture_rescued_images:
HashMap<u32, DrawableImage>` is mirrored as
`KmsCore.pictures: HashMap<u32, PictureRecord>` +
`RenderEngine.picture_paint: HashMap<u32, PicturePaintState>`.
The destruction ordering rule (decref the picture's drawable
backing only on `FreePicture`) is preserved.

### 3. Glyph-atlas per-glyph fence — drop the `queue_wait_idle`

v1's `GlyphAtlas::intern` (and `OpsStaging::ensure`, and several
glyph-pool drop paths) still calls `vkQueueWaitIdle`. The spec
flags this for Stage 3:

> Per-glyph queue_wait_idle in GlyphAtlas::intern — v1-era TODO.
> Gets folded into v2's RenderEngine rewrite during Stage 3.

In Stage 3a, the atlas-upload path routes through
`RenderEngine`'s submitted-op queue exactly like fill / put_image:

- `intern` takes the same `(key, w, h, pixels, …)`, but instead
  of `run_one_shot_op` (which submits + waits), it records the
  copy-buffer-to-image into a fresh CB and submits it via
  `PlatformBackend::submit_cb` against a `FenceTicket`.
- **Critical (codex round 1, point 1):** v1's `GlyphAtlas` holds
  a single persistent host-mapped staging buffer
  (`staging_buffer` / `staging_memory`) and `record_upload`
  copies into it then submits. v1 gets away with this because
  `run_one_shot_op` waits before returning — by the time the
  next `intern` overwrites the staging, the previous upload's
  CB has retired. **v2 cannot inherit that shape.** Without the
  per-upload wait, a back-to-back `intern(A)` + `intern(B)`
  would have B's memcpy land on top of A's still-pending GPU
  read, corrupting A's atlas slot. The v2 fix: **each upload
  owns its own staging bytes for the lifetime of its CB.**
  Concrete model:
  - `RenderEngine::intern_glyph` (the v2-side wrapper) does:
    1. Pack the glyph bytes into a slice borrowed from the
       engine's `BatchUploadArena` (allocates `w*h` bytes,
       returns `(vk::Buffer, u32 offset, host *mut u8)`).
    2. memcpy the glyph pixels into the arena slice.
    3. Record a fresh CB containing the layout transition +
       `vkCmdCopyBufferToImage(arena.buffer, atlas.image,
       region with arena.offset)` + back transition.
    4. Submit the CB via `PlatformBackend::submit_cb` against a
       new `FenceTicket`.
    5. Push the `SubmittedOp` (carrying the ticket AND the
       arena's per-batch slice handle, so the slice cannot be
       reused until the ticket signals).
  - The `BatchUploadArena` already retires its per-batch slices
    on ticket signal (Stage 2c pattern). The glyph upload
    becomes "a small slice from a big arena" rather than
    "a single mapped buffer mutated in place."
- The atlas image itself is a long-lived allocation owned by
  `RenderEngine` (not a `Drawable` in the store).
  **GPU execution ordering** between an atlas upload and a
  subsequent atlas-sampling paint comes from **same-graphics-queue
  submission order** — every atlas upload CB and every atlas-
  sampling paint CB (text run, render_composite_glyphs) submit
  on the engine's single graphics queue, and submission order
  on a single queue is an execution-order guarantee. `FenceTicket`
  is **not** a GPU wait primitive; it is the CPU-side lifetime
  ticket that gates resource recycling. The two are
  complementary: the queue gives ordering, the ticket gives
  retirement.
- For **CPU-side resource lifetime** the engine carries
  `atlas_last_upload_ticket: Option<FenceTicket>`. Each
  `intern` that records an upload replaces the slot with the
  new ticket. Every paint that samples the atlas (text run in
  3a, render_composite_glyphs in 3d) **clones
  `atlas_last_upload_ticket` onto its own SubmittedOp** so the
  arena slice holding the glyph bytes (and the atlas image
  itself) stay alive until both upload and consume submissions
  have retired on the CPU side. This is recycling correctness,
  not a GPU dependency claim.
- **If v2 ever splits to a separate transfer queue for atlas
  uploads** (Stage 5 perf work might), the upload→sample
  ordering above stops being a single-queue invariant and
  needs an explicit dependency: an exportable binary
  `vk::Semaphore` signaled by the upload submit and waited on
  by the sample submit, mirroring the `KmsSyncSemaphore`
  pattern used for KMS IN_FENCE_FD. Stage 3 stays single-queue
  to keep this implicit; multi-queue is risk-listed as a
  Stage 5 prerequisite.
- **Atlas-slot reuse on `FreeGlyphs` / `FreeGlyphSet`**: a
  glyph's atlas slot must not be overwritten or repacked while
  any paint that sampled it is still in flight. Stage 3 takes
  the simpler invariant: **freed slots are not reused within
  the same engine lifetime** (the shelf packer never reclaims).
  This matches v1's behaviour (the shelf packer is monotonic;
  the cache map's removed entries don't free the underlying
  pixels). Combined with the fixed 4096² atlas size, this is
  bounded — atlas-full triggers the §3a drop-the-glyph path
  before slot reuse pressure builds. Stage 5's grow + LRU
  pass owns slot reclamation and would track per-slot retired
  tickets to gate reuse safely.
- `intern` returns `Option<AtlasEntry>` synchronously (same
  contract as v1).
- `OpsStaging::grow` and `OpsStaging::Drop`'s `queue_wait_idle`
  paths in the atlas module **go away** as the atlas no longer
  owns a single persistent staging buffer. v1's
  `staging_buffer` / `staging_memory` / `staging_mapped` /
  `staging_size` fields are removed from `GlyphAtlas` in 3a.
  The v2 atlas owns the atlas image + view + memory + cache +
  shelves only.

**Test for the back-to-back race** (3a unit test, codex round 1
required): submit `intern(A)` and `intern(B)` without polling
the engine between calls (i.e. their CBs are both still in
flight when the test asks the GPU to sample both); record a
synthetic text run that samples A; record a second text run
that samples B; assert (via `get_image` on the destination
mirror) that A's pixels are intact AND B's pixels are intact.
A buggy shared-staging model would have B's memcpy clobber A's
in-flight upload bytes; the test fails loudly.

**Telemetry check:** with Stage 3a in, `vk_queue_wait_idle/s == 0`
even under text-heavy workloads. Stage 2 already enforces this
under non-text workloads; 3a extends the guarantee.

### 4. Clipping unification — GC clip + Picture clip + SHAPE

X11 has three layers of clipping that affect a paint op:

- **GC clip** (`set_clip_rectangles` / `set_clip_pixmap`) —
  applies to **Core ops only** (`poly_*`, `fill_*`,
  `image_text*`, `put_image`, `copy_*`). RENDER requests don't
  carry a GC; they reference Picture xids directly. Stage 3
  must **not** let the ambient `current_clip` (which v2 carries
  in `KmsCore.current_clip`) leak into protocol RENDER ops.
- **Picture clip** (`set_picture_clip_rectangles`) — applies
  to RENDER ops on a per-picture-record basis. The dst
  picture's `clip` + `clip_x`/`clip_y` define the effective
  scissor region.
- **SHAPE bounding region** — applies to scene composition (Stage
  4, deferred). Per § "Non-goals" §7. The Stage 3 risk this
  creates for the smoke matrix is handled in § "Risk 10"
  (shaped windows render inside their bounding rectangle, not
  the SHAPE outline).

Stage 3's `RenderEngine` clips at op-record time:

- **Core path:** `apply_clip_state` is already wired through
  `KmsBackendV2::apply_clip_state` (Stage 1b stub). Stage 3
  promotes the stub: takes `Option<ClipRectangles>`, stores in
  the engine's per-op clip state (single source of truth for
  the next Core-paint call), and the engine intersects on
  record. Core ops that v2 lowers internally to a RENDER draw
  (e.g. an `image_text*` background fill that goes through
  `render_fill_rectangles`) **pass the Core clip explicitly**
  into the engine call rather than relying on the picture
  record's clip — the GC clip and the picture clip are
  separate concerns and the engine call site is the only place
  with both in scope.
- **RENDER path:** `RenderEngine::render_composite` /
  `render_fill_rectangles` / `render_composite_glyphs` /
  `render_trapezoids` read the dst picture record's clip from
  `KmsCore.pictures` (only; no `current_clip` consultation)
  and intersect the destination region before recording.
- **Exact clip-rect scissoring (codex round 1, point 4):** v1's
  `build_render_composite_inputs` collapses the picture clip
  to a single bounding-box scissor (a union, not a true
  intersection). That over-paints holes in multi-rect clips —
  rendercheck spots this on its clip-region tests. **Stage 3
  v2 records one draw call per clip-rect intersection** (set
  scissor, draw, set next scissor, draw, ...) so multi-rect
  clips are honoured exactly. The per-rect-scissor cost is
  one extra `cmd_set_scissor` + descriptor-rebind per clip
  rect; clip-rect counts are typically <= 8 in practice; the
  cost is bounded.

The intersection helper is reused from v1's
`intersect_with_current_clip`; v2 carries an equivalent free
function in `kms::v2::clip` since v1's lives on `KmsBackend`. The
exact same implementation; small mechanical move. The
per-rect-scissoring helper is new code (v1's union shortcut is
the one place v2 deliberately diverges from v1).

### 5. Damage accumulation under RENDER

Stage 2 sets damage on the target storage at fill / put_image /
copy_area record time. Stage 3 extends the same hook to every new
paint primitive:

- Core text (image_text / poly_text): record damage at the union
  of the text bounding box AND the background fill rect (if
  image_text).
- RENDER ops (default rule): record damage at the destination
  rect intersected with the picture clip.
- Trapezoids / triangles (default rule): damage at the bounding
  box of all trapezoids' / triangles' extents intersected with
  the picture clip.
- **Op-aware override (codex round 1, point 8):** for PictOps
  that affect dst pixels even where the source / mask is zero —
  `Clear`, `Src`, `In`, `InReverse`, `Out`, `OutReverse`,
  `DisjointClear`, `DisjointSrc`, the various Conjoint*
  destructive ops — the damage region for a traps/triangles op
  is the **full destination clip region**, not the primitive
  bbox. Outside the primitive bbox, mask alpha is 0 but for
  these ops "mask=0 → dst becomes color × 0 = 0" still mutates
  dst. If we only damage the primitive bbox the scene won't
  repaint those zeroed pixels and the screen retains stale
  contents. The op-aware check is a small table lookup in the
  damage path. v1 has the same surface bug latent (its damage
  layer is per-window mirror sampled by scanout, so the bug is
  hidden); v2 must surface it correctly.
- Poly_* Core ops: damage at each rectangle's bounding box.
  Non-`GXcopy` GC.function paths (`GXclear`, `GXcopyInverted`,
  etc.) also need the op-aware override for the same reason —
  `GXclear` zeros every pixel in the GC's clip region whether
  the foreground bit is set or not. The damage region for
  these is the **full destination clip region**.

For each paint op, the **same two damage lists** from I5 are
written: protocol always, presentation iff
`scene_participating`. The snapshot/ack epoch from Stage 2 keeps
its semantics — paint between peek and ack survives. No model
changes; just more op sites that call `store.damage(id, rect)`,
plus the op-aware-override table for destructive ops.

### 6. PictFormat advertisement + RENDER version

v1 advertises a fixed list of RENDER picture formats via the
`render_pixmap_formats` reply (see `kms::render`). v2 inherits
the same list — RENDER protocol-level format negotiation
happens in the shared dispatcher, not in the backend, and the
backend only sees resolved `ynest_format` ids. v2 already
implements `render_format_for_ynest_id` (Stage 1b skeleton) by
delegating to `KmsCore`. No change needed at Stage 3 — formats
are protocol metadata.

**`render_query_version`** in v2 already returns `(0, 11)` from
Stage 1b (`backend.rs:1599`); v1 does the same. Stage 3 does
**not** treat version advertisement as a substage gate (codex
round 1, point 7) — the advertised version stays aligned with
v1 for client-compatibility, and `log_v2_gap` is the
enforcement mechanism for "this op isn't implemented yet."
Clients that probe version 11 then call CompositeGlyphs before
3d lands get a gap log and an empty reply, which is the same
shape they'd get if v2 advertised 10. Bumping or unbumping the
version mid-implementation churns the protocol surface for no
gain.

### 7. Open-question handling — `core.current_font` is per-GC?

v1's `core.current_font` is **single-valued** — one font xid
tracked for the whole backend. This is a v1-era simplification
that conflates "the most recently set GC font" with "the font
this paint should use". GTK + Cairo don't hit it because they
use RENDER glyphsets exclusively; Core-text apps like xterm hit
it because they only ever use one GC at a time. Stage 3 keeps
the same simplification — a real per-GC table is a Stage-3-or-
later cleanup but not load-bearing for any tested app. The
risk: a Core-text app that uses two fonts on two GCs (xosview?
xdaliclock?) would render with whichever font was last set.
Risk-list it (see § "Risks").

## Substage 3a — Glyph atlas + Core text

### Goal

Land the glyph atlas + text pipeline in `RenderEngine`, with
the v2 fence discipline (no `queue_wait_idle`). Wire the four
Core text ops + `open_font` / `close_font`.

### Scope of paint ops

- `open_font(name) -> (FontHandle, FontMetrics)` — same body as
  v1 (delegates to `KmsCore::font_loader.open_font`); `KmsCore`
  already owns `FontLoader` + `FontState`.
- `close_font(host_xid)` — remove from `KmsCore.fonts`.
- `image_text8(host_xid, fg, bg, len, body)` — draw `bg` rect
  spanning the text run, then draw text in `fg`.
- `image_text16` — same shape with CHAR2B.
- `poly_text8(host_xid, fg, body)` — TextItem8 sequence;
  delta-spaced; supports inline font-change items.
- `poly_text16` — same with CHAR2B + delta.

### Module shape

```rust
// crates/yserver/src/kms/v2/engine.rs (Stage 3a additions)

impl RenderEngine {
    /// One text run: render glyphs through FreeType (CPU-side),
    /// intern in atlas (each intern records a copy-buffer-to-image
    /// CB), record a single text-run draw on the target. Returns
    /// Err if any sub-step fails; caller logs + drops the run.
    pub(crate) fn image_text(
        &mut self,
        store: &mut DrawableStore,
        platform: &mut PlatformBackend,
        target: DrawableId,
        font_state: &FontState,
        foreground: u32,
        x: i32,
        y: i32,
        chars: &[char],
    ) -> Result<(), RenderError>;
}
```

`KmsBackendV2`'s `image_text8` / `image_text16` / `poly_text8` /
`poly_text16` resolve the font + foreground, decode the text-item
stream (poly_text), and call `engine.image_text` per run. v1's
decoder bodies in `backend.rs` migrate verbatim to `KmsBackendV2`
— the parsing logic doesn't depend on the rendering layer.

### Glyph-atlas fence integration

Per § "Cross-cutting" §3:

- `RenderEngine` owns `glyph_atlas: GlyphAtlas`. The atlas image
  is a private allocation, **not** a `Drawable` in the store.
- `RenderEngine` carries `atlas_last_upload_ticket:
  Option<FenceTicket>`. Each `intern` call that records an
  upload coalesces a new ticket into this slot.
- Every paint op that samples the atlas (text run, render
  composite_glyphs in 3d) clones `atlas_last_upload_ticket`
  into its `SubmittedOp.atlas_ticket` field so the GPU read
  cannot race the upload retirement, and the atlas image is
  guaranteed live until both upload and consume retire.

The upload CB itself is recorded inline (begin/transition/
copy_buffer_to_image/transition-back/end) and submitted via
`PlatformBackend::submit_cb`. The staging buffer is held by
the upload's `SubmittedOp` until its ticket signals; same
shape as Stage 2c's `put_image` staging.

**Atlas-full fallback:** v1 falls back to a pixman-CPU path
when the atlas is full. v2 doesn't bring pixman back. The
substitute (codex round 1 confirmed): at atlas-full, the
affected glyph is skipped (text advances by its
`character_width` but no glyph paint happens), the engine
logs `atlas_full` once per session, and the
`glyphs_dropped_atlas_full` lifetime counter increments per
dropped glyph. This is acceptable because (a) atlas-full has
never been observed in practice (4096×4096 R8 holds ~14k
typical glyphs), and (b) Stage 5's atlas grow/LRU work covers
the recovery path if it ever does come up.

### Scope of ops

- `render_create_picture(drawable, format, mask, values)` —
  inserts `KmsCore.pictures[xid] = PictureRecord::Drawable`,
  delegates to `render_change_picture` for the value-mask body.

  picture record.

  `DrawableStore`.

  `clip` + `clip_x` / `clip_y` on `PictureRecord::Drawable`.
- `render_set_picture_filter` — store filter; only Nearest
  honoured by RenderEngine in Stage 3.
- `render_set_picture_transform` — parse the 9-fixed-point
  matrix, store as `PictTransform`, normalise to affine for
  the shader.
- `render_create_solid_fill(color)` — premultiply + store
  `PictureRecord::SolidFill`.
- `render_create_linear_gradient` / `render_create_radial_gradient`
  — parse + store; gradient image generation lives in

  `KmsCore.glyphsets`. No GPU side until 3d.
- `render_free_glyphset` — drop record + 3d-allocated atlas
  reservations.

### RenderEngine adds

- `render_pipelines: Option<RenderPipelineCache>` — adopted
  from v1 wholesale (38 PictOps, lazy-built per
  `(op, dst_format, dst_has_alpha, component_alpha)`).
- `solid_src_image`, `solid_mask_image`, `white_mask_image` —
  per-engine 1×1 scratch images. Allocated lazily on first
  `render_composite` call (Stage 3c).
- `dst_readback: Option<DstReadback>` — scratch image used by
  the Disjoint/Conjoint shader-side blend paths. Allocated
  lazily.
- `picture_paint: HashMap<u32, PicturePaintState>` — owns
  gradient `GradientPicture` instances and any other
  GPU-residency picture state.
- `descriptor_arena: BatchDescriptorArena` — adopted from
  `kms::scheduler::batch_descriptor_arena`. Per-batch allocator
  for RENDER per-call descriptor sets. Resets when its owning
  SubmittedOp's ticket signals.
- **`drawable_view_cache:
  HashMap<(DrawableId, SamplerConfig, SwizzleClass), CachedDrawableView>`**
  (codex round 2). Same drawable sampled in different roles
  (source vs mask vs alpha-only) needs different sampler /
  swizzle bindings; keying on `DrawableId` alone would over-
  share. Defined classifications for Stage 3:
  - `SamplerConfig`: `{ filter: Nearest, address: Clamp |
    Repeat | MirroredRepeat | Border }` — the four cases
    Repeat-None/Normal/Pad/Reflect from the X11 RENDER picture
    record map onto. Nearest-only per Risk 6.
  - `SwizzleClass`: `{ RgbaIdent, AlphaOnlyR8, BgraIdent }` —
    three buckets covering the formats v2 supports:
    `R8_UNORM` masks read as `(0,0,0,r)` for the
    AlphaOnlyR8 case; `B8G8R8A8_UNORM` storage reads as
    `(b,g,r,a)` for BgraIdent; an RGBA pixmap (rare) reads as
    `(r,g,b,a)`.
  - **Eviction**: cache entries are dropped when the matching
    `Drawable` retires (`DrawableStore::poll_pending_retire`
    notifies the engine). No LRU; cache size is bounded by
    live drawable × ~6 entries which is comfortably under any
    descriptor budget.
  - **Telemetry**: `image_view_creates/s` (Stage 2 counter)
    must drop toward zero in steady state after warmup; a
    sustained non-zero rate means a cache miss pattern
    (typical: cache key too narrow, or eviction happening
    needlessly) — Stage 3f perf capture monitors this.

The shared move of `BatchDescriptorArena` + `BatchUploadArena`
to `kms::vk` happens here if it hasn't already (Stage 2 plan
called for it; verify on starting 3b — if v1 still references
`kms::scheduler::batch_*`, do the rename in a sibling
mechanical commit alongside 3b).

### Goal

backgrounds correctly; xfce panels show their chrome; MATE

### Scope

- `render_composite(op, src, mask, dst, src_x..h)` — full path
  through the v1-style pipeline + push-constant + per-rect
  draw, with **picture clip only** applied via per-rect
  scissor (per § Cross-cutting §4: protocol RENDER ops do not
  consult GC clip; only Core ops that lower internally to a

  v2 mirrors it.

### Glyphset format unpack

RENDER glyphsets carry one of three formats in practice:
`Pictformat_A8`, `Pictformat_A1`, `Pictformat_ARGB32`. (Codex
round 1, point 6: v1's `core.rs` knows A8/A1/ARGB32, not A4.
The A4 mention in the draft was wrong — no app emits A4 and
v1 doesn't decode it; v2 doesn't either.) All map to an `R8`
atlas slot in v2:

- **A8**: 1 byte per pixel, copied 1:1 into the atlas slot.
- **A1**: 8 pixels per byte. MSB-first per the X11 RENDER
  protocol (the bit for pixel `c` of row `r` lives at
  `byte = wire[r * stride + c/8]; bit = (byte >> (7 - c%8)) & 1`).
  The unpacker expands each bit to 0x00 / 0xFF in the atlas
  slot.
- **ARGB32**: 4 bytes per pixel (BGRA). The unpacker
  extracts the alpha byte only; the RGB is discarded (atlas
  is an alpha mask; the colour comes from the SolidFill
  source). This matches what `parse_add_glyphs` does in v1.

The v1 `kms::vk::ops::text` path already operates on R8 atlas
pixels; the same recorder shape applies. Only the unpack step
in `AddGlyphs` differs per format. **No per-format shelf
segregation** — the earlier draft saying "R8 / R8G8B8A8 / R1
packed via the same atlas" was a typo; everything lives in
one R8 atlas. (Codex round 1, point 6.)

### Scope of ops

- `poly_line(host_xid, fg, mode, points)` — coordinate-mode
  decoder, line rasterisation via filled-rectangle decomposition
  for axis-aligned segments + a thin-line shader for diagonals.

## Risks

**Risk 1 — `core.current_font` single-valued.** Per
§"Cross-cutting" §7. Apps that switch GCs with different fonts
mid-stream may render the wrong font. Mitigation: monitor
real-app smoke for this; if hit, promote to a `current_font_per_gc`
table (small migration). Codex round 1 confirmed deferral is
fine for Stage 3 (v1 has the same simplification). 3f real-app
smoke includes a manual two-GC-two-fonts assertion if cheap to
construct, but it's not a Stage-3-blocker.

**Risk 2 — Picture-clip + GC-clip interaction.** Protocol
RENDER ops use the dst picture's clip only; Core ops use the
GC clip only (per § Cross-cutting §4). Edge case: an app that
paints to a backing pixmap through both Core ops AND through
a RENDER picture wrapping the same pixmap expects each path
to honour its own clip independently. v1 has the same shape;
v2 mirrors it. Risk is wire-bug, not model-bug; mitigated by
rendercheck + the 3c tests `render_composite_picture_clip_per_rect`
and `render_composite_no_gc_clip_leak` (which together
exercise both halves of the clip-domain separation).

**Risk 3 — `dst_readback` allocation race under
Disjoint/Conjoint.** The shader-side blend path reads the
dst pixel via a sampled image bound to `dst_readback`. v1
allocates one `DstReadback` per backend (single source for
all in-flight composites). v2 must hold the readback CB's
ticket on the dst's `last_render_ticket` so a subsequent
paint can't free the dst storage mid-Disjoint blend. Already
implied by the `touch_render_fence` rule, but adding a
unit test (`disjoint_blend_holds_dst_fence_alive`) is cheap
insurance.

**Risk 4 — Glyphset format A1 unpack endianness.** RENDER
spec is unambiguous (MSB-first per byte), but the v1 code
path has a known endian-sensitive comment in the unpacker.
v2's unpacker is byte-identical to v1; the risk is a v1
latent bug that v2 inherits. Mitigation: rendercheck covers
A1 glyphsets; if v1 passes, v2 passes.

**Risk 5 — Atlas-full recovery.** Resolved (codex round 1):
drop the glyph + log + counter. No pixman re-introduction.

**Risk 6 — `render_set_picture_filter` bilinear.** Resolved
(codex round 1): defer to Stage 5, per spec. Sampler-selection
+ descriptor cache keys + tests + rendercheck diffs are more
than the "one line + fragment-shader change" the draft
claimed. Nearest-only stands for Stage 3.

**Risk 7 — Real-app smoke matrix breadth.** The list in §3f
covers GTK + Qt-via-Xrender + Cairo + Pango + Xt-applications
(xterm + xclock + xeyes + xfd) + the SHAPE delta note for
xeyes. Deliberately omits Compiz (Stage 5 territory; Compiz
uses GLX_EXT_texture_from_pixmap which v2's RENDER path
doesn't engage) and dwm / herbstluftwm (Wayland-ish lineage;
nothing v2-specific). If codex wants additional apps in the
matrix, name them.

**Risk 8 — fuji v1 baseline drift.** The perf gates compare
against v1 captures. If the v1 baseline is not captured fresh
in the same session as the v2 capture, micro-environment
factors (theme version, screen contents, recent kernel
upgrade) can move the v1 baseline 10–15%. Mitigation:
**capture v1 and v2 in the same hardware-smoke session, no
reboot, same theme, same starting client load**. The 3f
acceptance includes the v1 baseline capture as a *prerequisite
step*, not "compare against an older capture."

**Risk 9 — CompositeGlyphs coverage gap (codex round 1).**
Stage 3d v1-parity scope: `Over` + `SolidFill` source + A8 /
A1 / ARGB32-as-A8 only. Other op / source combinations log a
gap and skip. Apps that use a non-Over PictOp with glyphs
(rare in practice — GIMP's text-stamping with subtract /
multiply ops is the only example I know of) won't render
correctly. Apps that use a Drawable or Gradient source with
glyphs (e.g. a gradient-filled text effect in a Cairo SVG)
also won't. v1 has the same limitation; v2 inherits it. A
follow-up "CompositeGlyphs full coverage" plan after Stage 3
close lifts this — moving from `TextPipeline` to the full
`RenderPipelineCache` for glyph composites — but that's
explicitly post-Stage-3 work.

**Risk 10 — SHAPE-output-clip deferral (codex round 1).**
Stage 4 wires SHAPE-bounded scene clipping. In Stage 3,
shaped windows (xeyes, oclock, xclock-with-`-fg`, anything
that calls `set_shape_rectangles`) render inside their
rectangular bounds — the "corners outside the SHAPE outline"
show whatever was beneath (root pixel) rather than being
masked away. The smoke matrix calls this out as the only
known v2 visual delta vs v1 acceptable for Stage 3 close.
Risk: if a real app's correctness depends on the SHAPE
outline (e.g. a panel that uses SHAPE to define its hit
region — input-shape side is handled by `KmsCore.shape_input`
and works correctly because input dispatch reads it
directly), Stage 3 might present a worse UX than v1. Tracked
for Stage 4.

**Risk 11 — Adapter complexity.** §"Cross-cutting" §1
promises a "mechanical glue" `DrawableImageRef` adapter.
Codex round 1 flagged that the v1 recorders may want **more
than the five raw image fields** — specifically, the RENDER
recorder consults source picture format swizzles, mask
absence, and whether the dst has an alpha channel. v2's
`Drawable` carries `format` (`vk::Format`), which covers
swizzle; the mask-absence path takes a `None` for the mask
descriptor binding (v1 uses the `white_mask_image` scratch
to fill that slot — v2 mirrors that exactly); the
dst-has-alpha flag is derivable from `Drawable.format`. If
3a's text path ends up with type-system fights past ~50 lines
of adapter code, revisit and grow a v2-native recorder
instead of reusing v1's.

**Risk 12 — `OpsStaging` / `vk::ops::run_one_shot_op` Drop
paths still call `queue_wait_idle` (codex round 1).** The
broader engine-init / shutdown / pool-grow paths in
`kms::vk::ops::mod.rs` and `kms::vk::pixmap_pool.rs` still
have `queue_wait_idle` in their Drop implementations. v2's
hot-path commitment is "zero `queue_wait_idle` in **steady
state**"; teardown / drop-time waits are acceptable. The risk
is that v2 inherits a code path where a `run_one_shot_op` is
still on a hot route — verify in 3a by grepping
`run_one_shot_op` call sites that v2's `RenderEngine`
references and ensure none of them sit on a per-frame /
per-glyph path. The `intern_glyph` rewrite in 3a removes the
big one; the rest are explicitly off-hot-path.

**Risk 13 — Self-composite aliasing (codex round 1).** RENDER
`Composite` can have `src == dst` (or `src` aliasing some
backing pixel that `dst` writes). v1 has a "self-composite"
codepath that detects this and routes through a scratch
image (see `kms::backend.rs`'s copy_area same-image
overlap handling, which is structurally similar). v2's
`render_composite` must mirror that — same dst + src
DrawableId triggers a copy-out via a scratch image into the
arena, then composite from the scratch to dst. Without this,
the GPU samples the dst as it's being written and the
result is undefined. Unit test: `render_composite_self_alias`.

**Risk 14 — `planemask` correctness (codex round 1).**
Non-`0xFFFFFFFF` planemask on a depth-32 drawable should
preserve unmasked bits of the dst. v1 implements this via
`LogicFillPipeline`. Stage 3f adopts the same path. Risk: the
RENDER paint paths (3c) **do not** honour planemask — they
go through the standard `RenderPipelineCache` which has no
planemask logic. This matches v1 (RENDER ops bypass GC
state). The Core ops in 3f that go through `logic_fill_pipeline`
honour it. The risk is incomplete documentation: spec it
clearly so codex / future maintainers don't expect RENDER
planemask support.

**Risk 15 — Adapter view caching for masks.** Resolved (codex
rounds 1+2). 3b builds the cache keyed on
`(DrawableId, SamplerConfig, SwizzleClass)` with the three
`SwizzleClass` buckets enumerated in § 3b "RenderEngine adds".
Telemetry hit-rate visibility comes from `image_view_creates/s`
(must trend to zero in steady state). Cache eviction follows
the Drawable's retirement; no separate LRU.
